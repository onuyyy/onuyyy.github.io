## 0. 출발점 — 왜 이 얘기가 나왔나

권한 체크 흐름에서:
- 로그인 시 권한(역할·메뉴·화면)을 `LoginUser` 객체에 담아 **서버 세션(메모리)**에 보관
- 매 요청마다 권한 검사 = **DB 재조회가 아니라 세션(메모리) 조회** → `map.get()` 한 번
- 그래서 "매번 서버가 검사해도 안 느리다"

즉 **세션은 일종의 서버측 캐시**다. 여기서 "메모리가 왜 빠른가?"로 확장.

## 1. 메모리 계층 (Memory Hierarchy)

### 왜 계층이 존재하나?
세상에 **"빠르면서 + 크면서 + 싼"** 저장장치는 없다. 셋 중 둘만 고를 수 있다.

```
빠름  ↔  쌈/큼   은 서로 반대.
빠른 건 비싸고 작다.  싼 건 크고 느리다.
```

그래서 **빠르고 작은 걸 CPU 가까이, 느리고 큰 걸 멀리** 계단식으로 배치한 것.

### 속도 계층 (빠름 → 느림)

| 순위 | 저장 위치 | 예시 | 속도 | 네트워크? | 디스크? |
|------|-----------|------|------|-----------|---------|
| 1 | CPU 레지스터 | 하드웨어 | 1 사이클 | ❌ | ❌ |
| 2 | CPU 캐시 (L1/L2/L3, SRAM) | 하드웨어 | 1~40 사이클 | ❌ | ❌ |
| 3 | RAM / 메인메모리 (DRAM) | **세션(LoginUser)**, HashMap, Caffeine 로컬캐시 | 수백 사이클 | ❌ | ❌ |
| 4 | 원격 캐시 | Redis, Memcached | 마이크로~밀리초 | ✅ (1번 왕복) | ❌ (RAM 저장) |
| 5 | DB | MySQL, Oracle | 밀리초~ | ✅ | ✅ |
| 6 | 디스크 (SSD/HDD) | 파일시스템 | 0.1~10밀리초 | (로컬) | ✅ |

```
        ┌─ 빠름 / 작음 / 비쌈
   CPU ─┤  레지스터       (몇 바이트)
        │  L1 캐시        (KB)       ← SRAM
        │  L2 / L3 캐시   (MB)       ← SRAM
        │  RAM            (GB)       ← DRAM (= 세션이 여기)
        │  SSD / 디스크   (TB)
        └─ 느림 / 큼 / 쌈   네트워크(DB, 다른 서버)
```

## 2. "가까울수록 빠른" 이유 3가지

### 이유 ① 물리적 거리 (빛의 속도 한계)

- 전기 신호도 빛의 속도를 못 넘는다. 빛은 **1나노초에 약 30cm** 이동.
- CPU는 3GHz = 1초에 30억 번 = **1사이클 ≈ 0.33나노초**.
- 그 짧은 시간에 신호가 갈 수 있는 거리는 **약 10cm**.

| 저장소 | 물리적 위치 | 거리 | 비용 |
|--------|-------------|------|------|
| 레지스터 / L1 | CPU 칩 **안** | 0.1~1mm | 1~4 사이클 |
| RAM | 메인보드 위 | 5~10cm | 수백 사이클 (대기) |
| DB (다른 서버) | 네트워크 너머 | 수십 km | 수백만 사이클 |

→ **멀면 신호 왕복 동안 CPU가 논다.** 가까울수록 빠른 건 결국 거리 문제.

### 이유 ② 부품 자체가 다름 (SRAM vs DRAM)

| | CPU 캐시 | RAM (메인메모리) |
|---|---|---|
| 부품 | **SRAM** | **DRAM** |
| 1비트 구성 | 트랜지스터 6개 | 트랜지스터 1개 + 축전기 1개 |
| 속도 | 매우 빠름 | 느림 |
| 가격/용량 | 비싸고 작음 (MB) | 싸고 큼 (GB) |
| 특징 | 전기 계속 흐름 | 주기적 충전(refresh) 안 하면 데이터 날아감 |

- **SRAM**: 트랜지스터 6개로 비트를 "꽉 잡고" 있어 즉시 읽힘. 비싸고 자리 많이 먹음 → 캐시는 작다(MB).
- **DRAM**: 축전기에 전하 저장. 싸고 빽빽 → 용량 큼(GB). 충전 상태 확인 후 읽어야 해서 느림.

→ **캐시가 빠른 건 "비싼 부품(SRAM)을 CPU 옆에 조금만" 박았기 때문.**

#### L1 / L2 / L3 캐시는 전부 SRAM

CPU 칩 안의 캐시는 셋 다 SRAM이다. 같은 부품인데 **크기·거리** 때문에 속도가 갈린다.

| | 부품 | 크기 | 거리(코어 기준) | 속도 |
|---|---|---|---|---|
| 레지스터 | SRAM 유사(별도 회로) | 몇 바이트 | 코어 안 | 제일 빠름 |
| L1 | SRAM | 작음 (KB) | 제일 가까움(코어마다) | ~1ns |
| L2 | SRAM | 중간 | 조금 멀음(코어마다) | ~4ns |
| L3 | SRAM | 큼 (MB) | 멀음(여러 코어 공유) | ~10ns+ |

→ 같은 SRAM이라도 **크면 자리를 많이 차지 → 신호 거리가 길어짐 → 느려짐** ("거리 = 속도" 원리).
→ 그래서 작고 빠른 L1은 코어 바로 옆, 크고 느린 L3는 떨어뜨려 공유하는 계단 구조.

```
DRAM 부품 = 메인메모리(RAM 16GB), Redis 데이터
SRAM 부품 = L1, L2, L3 캐시 (전부) + 레지스터(유사 회로)
```

> ※ RAM = "임의 접근 메모리(Random Access Memory)"라는 **큰 범주**.
> 그 안에 SRAM(캐시용) + DRAM(메인메모리용)이 있다.
> 일상에서 "램 16GB"라고 하면 거의 **DRAM(메인메모리)**을 가리킴.
> 캐시(SRAM)는 "램"이라 안 부르고 "캐시"라고 따로 부름. 둘 다 기술적으론 RAM.
>
> ※ "캐시"는 부품이 아니라 "역할"(자주 쓰는 걸 더 빠른 곳에 임시 복사)이다.
> CPU 캐시만 SRAM(부품)이고, **Redis/세션 캐시는 역할만 캐시일 뿐 부품은 그냥 DRAM(일반 램). SRAM 아님.**

### 이유 ③ 지역성 (Locality of Reference) — 캐시가 작아도 효과 있는 이유

프로그램은 **같은/근처 데이터를 또 쓰는 경향**이 있다:

- **시간 지역성(Temporal)**: 방금 쓴 데이터는 곧 또 쓴다 (반복문 변수 `i`)
- **공간 지역성(Spatial)**: 쓴 데이터 *옆*도 곧 쓴다 (배열 `arr[0]` → `arr[1]`)

그래서 CPU는 RAM에서 데이터를 가져올 때 **주변까지 한 덩어리(보통 64바이트 = "캐시 라인")로** 끌어와 캐시에 둔다.

```
arr[0] 요청 → 캐시에 없음(miss) → RAM에서 arr[0]~arr[7] 통째로 가져옴
arr[1] 요청 → 캐시에 있음(hit!) → 즉시 반환   ← RAM 안 감
```

- **캐시 히트(hit)**: 캐시에 있어서 바로 꺼냄
- **캐시 미스(miss)**: 없어서 더 느린 계층까지 가야 함
- 히트율이 높아서 캐시가 작아도 평균 속도가 크게 빨라짐.

## 3. 속도 체감 — 사람 시간으로 환산

CPU 1사이클을 **1초**로 뻥튀기하면:

| 저장소 | 실제 시간 | 사람 체감 (1사이클=1초) |
|--------|-----------|--------------------------|
| L1 캐시 | ~1ns | **1초** |
| L2 캐시 | ~4ns | 4초 |
| RAM | ~100ns | **약 2분** |
| SSD | ~0.1ms | 1.5일 |
| 디스크(HDD) | ~10ms | **5개월** |
| 네트워크(DB 왕복) | 수 ms+ | 수개월 |

→ **L1캐시(1초) vs RAM(2분)** 차이가 이만큼.
→ RAM도 CPU 입장에선 "느린 편" → 그래서 캐시 히트가 중요.
→ DB(수개월)는 RAM(2분)보다 수만 배 느림 → 권한 체크를 DB 대신 세션에서 하는 이유.

## 4. 저장소 = 2개의 축으로 봐야 한다

가장 헷갈리는 포인트. **"메모리에 있다" ≠ "내 서버에 있다".** 다른 축이다.

```
축①  "데이터를 어디에 저장하나"  →  RAM(메모리)  vs  디스크
축②  "그 프로그램이 어디서 도나"  →  내 서버 안   vs  다른 서버(네트워크)
```

| | 저장 방식 (축①) | 위치 (축②) |
|---|---|---|
| 세션 (LoginUser) | **RAM** | **내 서버 안** |
| **Redis** | **RAM** | **다른 서버** |
| DB (MySQL/Oracle) | 디스크 | 다른 서버 |

→ "Redis는 메모리에 있다"(축①)도 맞고 "Redis는 별도 서버다"(축②)도 맞다. **모순 아님.**
→ Redis = *데이터는 RAM에 두지만, 그 RAM은 다른 컴퓨터의 RAM*.


### 데이터 가지러 갈 때 "건너는 장벽" 개수

| 대상 | 네트워크 건넘? | 디스크 건드림? | 결과 |
|------|----------------|----------------|------|
| 세션 (내 RAM) | ❌ | ❌ | 🟢 제일 빠름 |
| Redis | ✅ (1번) | ❌ (RAM 저장) | 🟡 중간 |
| DB | ✅ (1번) | ✅ (디스크) | 🔴 제일 느림 |

→ Redis가 빠른 핵심: **"별도 서버지만 데이터를 디스크 아니라 RAM에 들고 있어서."**

## 6. Redis = "RAM에 데이터를 두는 DB서버"

```
일반 DB (MySQL/Oracle)  =  별도서버  +  데이터를 💾디스크에 저장
Redis                   =  별도서버  +  데이터를 🧠RAM에 저장
                           └─같음─┘      └────여기만 다름────┘
```

- "별도 서버"라서 → DB처럼 IP·포트로 접속, 네트워크 건너감
- "RAM 저장"이라서 → 디스크 안 뒤져 DB보다 빠름
- 공식 명칭: **인메모리 데이터베이스(In-Memory Database)**

### RAM 저장의 트레이드오프

| | 디스크 DB | Redis (RAM) |
|---|---|---|
| 속도 | 느림 | **빠름** |
| 전원 꺼지면 | 데이터 남음 (안전) | 날아감 (휘발성) ⚠️ |
| 용량 | 큼 (TB 단위 싸게) | 작음 (RAM 비쌈, GB 단위) |

- Redis 용도: 세션, 캐시, 임시 토큰, 순위표, 조회수 카운터 등 (날아가도 되는/사본 데이터)
- 진짜 원본(주문, 계좌)은 → 디스크 DB
- ※ Redis도 옵션으로 디스크 백업(persistence) 가능. 단 평소 본진은 RAM.

## 7. DB를 같은 서버 vs 별도 서버 — 어디가 빠른가?

### 단건 쿼리 속도만 보면 → 같은 서버가 약간 빠름 (네트워크 절약)

```
[같은 서버]  앱 → DB:  loopback(자기 호출) ≈ 0.05ms   ← 네트워크 사실상 0
[별도 서버]  앱 → DB:  같은 데이터센터 LAN  ≈ 0.1~0.5ms ← 네트워크 비용 발생
```

### 그런데 실무는 거의 다 분리한다

| 분리 이유 | 한 서버에 몰면 |
|-----------|----------------|
| **자원 경쟁** | 웹이 CPU/RAM 많이 쓰면 DB가 굶음. 디스크 I/O 싸움 → 둘 다 느려짐 |
| **확장(scale)** | 웹 서버 늘리려는데 DB가 붙어있으면 같이 복제 불가 |
| **장애 격리** | 서버 1대 죽으면 웹·DB 동시 다운 |

### 반전 — 네트워크 비용은 생각보다 작다

```
쿼리 1건 총 10ms 라면:
  네트워크 왕복      0.3ms   ← 별도서버라 추가 (3%)
  쿼리 파싱/실행     2ms
  디스크/버퍼 읽기   7ms     ← 진짜 병목
```

→ 병목은 디스크·쿼리(7ms)지 네트워크(0.3ms) 아님.
→ 네트워크 없애도 10ms → 9.7ms 정도. 이득 작음.
→ 반면 자원 경쟁으로 잃는 게 더 큼.

### 비교 그림

```
[A] 한 서버 (웹+DB)            [B] 분리 (실무 표준)
┌──────────────────┐          ┌─────────┐   ┌─────────┐
│  🖥️ 서버 1대      │          │🖥️ 웹서버 │   │🖥️ DB서버│
│  ┌────┐  ┌────┐  │          │ ┌────┐  │   │ ┌────┐  │
│  │ 웹 │←→│ DB │  │   LAN    │ │ 웹 │──┼───┼→│ DB │  │
│  └────┘  └────┘  │  ───────  │ └────┘  │0.3│ └────┘  │
│  💾 디스크 공유    │          │         │ms │ 💾 전용  │
│  (자원 뺏음)      │          │(자원독립)│   │(자원독립)│
└──────────────────┘          └─────────┘   └─────────┘
 네트워크 0, 근데              네트워크 0.3ms,
 자원경쟁·확장불가·동반사망     독립확장·격리 ✅
```

→ 한 줄: **"한 대에 몰면 한 쿼리는 살짝 빠르지만, 전체 시스템은 느리고 약하다."**
→ 소규모/개발용 = 한 대(편함), 운영 = 분리(안정·확장).

## 8. 핵심 요약

| 주제 | 한 줄 정리 |
|------|-----------|
| 메모리 계층 | 빠르고 작은 걸 CPU 가까이, 느리고 큰 걸 멀리. 트레이드오프(속도↔용량/가격) |
| 캐시가 빠른 이유 | ① 거리(빛속도) ② 부품(SRAM) ③ 지역성(미리 끌어옴) |
| SRAM vs DRAM | SRAM=캐시(빠름/작음), DRAM=메인메모리(느림/큼). 둘 다 RAM |
| 2개의 축 | "RAM에 저장" ≠ "내 서버에 있음". 다른 축 |
| 세션이 빠른 이유 | 내 프로세스 RAM → 네트워크 0, 디스크 0 |
| Redis | RAM에 데이터를 두는 별도 DB서버 (인메모리 DB). 빠르지만 휘발성/소용량 |
| DB 위치 | 같은 서버=단건 약간 빠름. 근데 자원경쟁·확장·격리 때문에 운영은 분리 |

### 속도 순서 (최종)
```
CPU캐시(SRAM) > RAM/세션(DRAM) > 원격캐시(Redis) > DB(디스크)
└─── 내 프로세스 안 ───┘      └─── 네트워크 너머 ───┘
결정적 차이 = "네트워크/디스크를 건너느냐"
```

### 응용 (소프트웨어 캐싱의 원리)
세션캐시 · Redis · CDN 다 똑같은 원리:
**"자주 쓰는 걸 더 가까운(빠른) 층에 복사해두자."**
