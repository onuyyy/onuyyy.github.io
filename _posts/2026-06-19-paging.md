## 1. LIMIT 문법

`LIMIT`은 "정렬된 결과에서 일부만 잘라내는" 명령이다.

### 형식 ① 개수만
```sql
LIMIT {size}
```
```sql
select * from tbl_board order by bno desc limit 10;   -- 앞에서 10개
```

### 형식 ② offset + 개수
```sql
LIMIT {offset}, {size}
```
```sql
select * from tbl_board order by bno desc limit 15, 10;  -- 15개 건너뛰고 10개
```

### 형식 ③ OFFSET 키워드 (읽기 명확, 권장)
```sql
LIMIT {size} OFFSET {offset}
```
```sql
select * from tbl_board order by bno desc limit 10 offset 15;  -- ②와 동일
```
> ②는 `offset, size` 순서, ③은 `size offset offset` 순서라 **숫자 위치가 반대**다. 헷갈리면 ③을 써라.
> ②③ 둘 다 MySQL/MariaDB에서 동작한다. (PostgreSQL은 ③ 스타일)

### ⚠️ order by는 사실상 필수
`LIMIT`은 "정렬된 결과의 일부"를 의미한다. **order by가 없으면 DB가 어떤 순서로 줄지 보장하지 않는다** → 페이지마다 글이 뒤죽박죽 될 수 있다. 페이징엔 항상 정렬 기준(보통 PK `bno`)을 명시한다.

```sql
-- 게시판은 보통 최신 글이 위로 → bno desc
select * from tbl_board order by bno desc limit 10;
```

## 2. offset 이해

**offset = 앞에서 건너뛸(skip) 행의 개수.** 0부터 센다.

```
정렬된 전체 행:
[0] [1] [2] ... [14] [15] [16] ... [24] [25] ...
 └──── 건너뜀(skip) ────┘ └──── 가져옴 size개 ────┘
        offset = 15              size = 10
```

- `offset 0` = 맨 처음부터 → 그래서 **offset을 생략하면 0**으로 동작한다.
  ```sql
  limit 10        -- = limit 0, 10  (1페이지)
  ```
- offset은 0부터 세므로 `offset 15`는 **16번째 행부터** 가져온다.

## 3. 페이지 번호 → offset 변환

화면에서 받는 건 **페이지 번호(page)** 와 **페이지당 개수(amount)**. 이걸 SQL의 offset으로 바꿔야 한다.

```
offset = (page - 1) * amount
```

| page | amount | offset | SQL |
|------|--------|--------|-----|
| 1 | 10 | (1-1)×10 = 0  | `limit 0, 10`  |
| 2 | 10 | (2-1)×10 = 10 | `limit 10, 10` |
| 3 | 10 | (3-1)×10 = 20 | `limit 20, 10` |

> 2페이지 = "1페이지 분량(앞 10개)을 건너뛰고 그 다음 10개" → offset 10.

### ⚠️ LIMIT엔 연산식을 못 넣는다
```sql
-- ❌ 안 됨: LIMIT은 상수/바인딩 값만 받음
limit (page-1)*amount, amount
```
→ **계산은 Java(또는 MyBatis)에서 미리 해서 값으로 넘긴다.**

## 4. 카운트와 전체 페이지 수

### 카운트는 왜 필요한가?
LIMIT로 가져온 데이터만으론 **"전체 페이지가 몇 개인지" 알 수 없다.** 지금 보는 10개가 전부인지, 뒤에 9990개가 더 있는지 모른다.
→ **전체 건수를 따로 세는 쿼리**가 필요하다.

```sql
select count(*) from tbl_board where bno > 0;
```

### 전체 페이지 수 = 올림 나눗셈(ceil)
```
totalPage = ceil(total / amount)
```
| total | amount | total/amount | totalPage |
|-------|--------|--------------|-----------|
| 100   | 10     | 10.0         | 10 |
| 101   | 10     | 10.1 → 올림  | 11 |
| 95    | 10     | 9.5 → 올림   | 10 |

> 101건이면 100건 채운 10페이지 + 남은 1건짜리 11페이지 → **올림**이 필요.

Java로 ceil 구현:
```java
int totalPage = (int) Math.ceil(total / (double) amount);
// 정수 연산만으로:
int totalPage = (total + amount - 1) / amount;
```

## 5. 화면 페이지 블록 만들기

전체가 1000페이지여도 화면엔 `[이전] 21 22 ... 30 [다음]`처럼 **10개씩 블록**으로 보여주는 게 보통이다.

```java
int page   = 23;     // 현재 페이지
int amount = 10;     // 페이지당 글 수
int total  = 1234;   // count(*) 결과

// 현재 페이지가 속한 블록의 끝 페이지 (블록 크기 10)
int endPage   = (int) (Math.ceil(page / 10.0)) * 10;   // 23 → 30
int startPage = endPage - 9;                            // 21

// 실제 마지막 페이지
int realEnd = (int) Math.ceil(total / (double) amount); // 124

// 마지막 블록 보정: 끝 페이지가 실제보다 크면 잘라줌
if (endPage > realEnd) endPage = realEnd;

boolean prev = startPage > 1;       // [이전] 버튼 표시?
boolean next = endPage < realEnd;   // [다음] 버튼 표시?
```

계산된 `startPage~endPage`, `prev`, `next`를 JSP로 넘겨 페이지 번호를 그린다.

```jsp
<c:if test="${prev}"><a href="?page=${startPage-1}">이전</a></c:if>
<c:forEach var="p" begin="${startPage}" end="${endPage}">
  <a href="?page=${p}">${p}</a>
</c:forEach>
<c:if test="${next}"><a href="?page=${endPage+1}">다음</a></c:if>
```

## 6. 성능: LIMIT의 한계와 해결

### 문제: deep paging (offset이 깊을수록 느려짐)
`limit 90000, 10`은 DB가 **앞 90000건을 실제로 읽어서 버리고** 그 다음 10건만 돌려준다. → offset이 클수록 읽고 버리는 양이 늘어 느려진다.

```sql
select * from tbl_board order by bno desc limit 90000, 10;
-- 90010건 읽어서 90000건 버림 = 낭비
```

| offset | 읽는 양 | 체감 속도 |
|--------|---------|-----------|
| 0      | 10      | 빠름 |
| 1000   | 1010    | 보통 |
| 100000 | 100010  | 느림 |

### 전제: 인덱스
`order by`·`where` 컬럼에 **인덱스**가 있어야 한다. PK(`bno`)는 자동으로 인덱스가 있으므로 `order by bno`는 정렬 비용이 거의 없다. (인덱스 없는 컬럼으로 정렬하면 전체 정렬이라 더 느림)

### 해결책 ① 커버링 인덱스 + 조인 (offset 방식 유지하면서 개선)
필요한 컬럼을 다 읽지 말고, **PK만 먼저 페이징**한 뒤 본문을 join한다.
```sql
select b.*
from tbl_board b
join (
  select bno from tbl_board order by bno desc limit 90000, 10
) t on b.bno = t.bno
order by b.bno desc;
```
서브쿼리는 인덱스(bno)만 훑어서 90000건 스킵 → 본문(TEXT 등)은 10건만 읽음 → 빨라짐.

### 해결책 ② 커서(keyset) 페이징 (근본 해결)
"몇 번째부터"가 아니라 **"마지막으로 본 bno 다음부터"** 가져온다.
```sql
-- 첫 페이지
select * from tbl_board order by bno desc limit 10;

-- 다음 페이지: 직전 페이지 마지막 bno가 91이었다면
select * from tbl_board
where bno < 91          -- 인덱스로 바로 점프 → 스킵 비용 없음
order by bno desc
limit 10;
```

| 방식 | 임의 페이지 점프("5페이지로") | 깊은 페이지 성능 | 용도 |
|------|------------------------------|------------------|------|
| `limit offset, size` | 가능 | 느려짐 | 번호형 페이징(일반 게시판) |
| keyset (`where bno < ?`) | 어려움 | 항상 빠름 | 무한스크롤·대용량 |

> **실무 감각:** 수천~수만 건 게시판은 그냥 offset 방식으로 충분하다(인덱스만 있으면 OK). deep paging 최적화는 데이터가 매우 많고 뒷페이지까지 자주 가는 서비스에서만 신경 쓰면 된다. **측정 없이 미리 최적화하지 말 것.**

## 7. 커서(keyset) 페이징 자세히

> SNS 피드·무한스크롤·대용량 목록의 표준 방식. offset 방식의 deep paging 성능 문제를 근본적으로 없앤다.

### 7-1. 핵심 아이디어
offset 방식은 **"앞에서 N개 건너뛰고"** 라서, 건너뛸 N개를 매번 읽어서 버린다(느림).
커서 방식은 **"마지막으로 본 값보다 작은(또는 큰) 것부터"** 라서, 인덱스로 그 지점에 바로 점프한다(빠름).

```
offset:  [버림][버림]...[버림] [10개 줌]      ← 버리는 비용이 깊을수록 커짐
cursor:  ────인덱스로 점프───→ [10개 줌]      ← 깊이와 무관하게 일정
```

### 7-2. 기본 SQL

```sql
-- 첫 페이지 (커서 없음)
select * from tbl_board
order by bno desc
limit 10;

-- 다음 페이지: 직전 페이지의 마지막 bno(=lastBno)를 받아서 그보다 작은 것부터
select * from tbl_board
where bno < #{lastBno}      -- desc니까 "더 작은 bno" = 다음 글들
order by bno desc
limit 10;
```

- `bno`는 PK라 인덱스가 있으므로 `where bno < ?`는 인덱스 탐색으로 즉시 위치를 찾는다 → offset 스캔 없음.
- 정렬 방향과 비교 부호를 맞춰야 한다:
  - `order by bno desc` → `where bno < lastBno` (작아지는 방향)
  - `order by bno asc`  → `where bno > lastBno` (커지는 방향)

### 7-3. "커서"가 뭔가?
**다음 페이지를 어디서부터 가져올지 가리키는 기준값.** 여기선 직전 페이지의 마지막 `bno`.
- offset 방식은 클라이언트가 "3페이지 줘"라고 함 → 서버가 `(3-1)*10` 계산.
- 커서 방식은 클라이언트가 "bno 91 다음부터 줘"라고 함 → 서버가 `where bno < 91`.

서버 응답에 **다음 커서**를 같이 내려주는 게 보통이다.
```json
{
  "content": [ ...10개... ],
  "nextCursor": 81,        // 이 페이지 마지막 bno → 다음 요청 때 이걸 보냄
  "hasNext": true
}
```
`hasNext` 판단 트릭: **limit을 +1 해서(11개) 조회** → 11개가 오면 다음 페이지 있음, 10개만 잘라서 응답하고 11번째의 bno를 커서로 쓴다.

```sql
... order by bno desc limit 11;   -- 1개 더 가져와서 hasNext 판정
```

### 7-4. 단점과 주의점
- **임의 페이지 점프 불가.** "갑자기 50페이지로" 가 안 된다. 앞/뒤 순차 이동(무한스크롤·더보기)에만 적합.
- **정렬 기준이 유일(unique)해야 안전.** `bno`(PK)처럼 중복 없는 값이어야 경계에서 누락/중복이 안 생긴다.
  - `regDate`처럼 **중복 가능한 값**으로 정렬하면, 같은 시각 글이 경계에 걸쳐 잘릴 수 있다.
  - 해결: **(정렬키, PK) 복합 커서**로 tie-break.
    ```sql
    where (regDate < #{lastDate})
       or (regDate = #{lastDate} and bno < #{lastBno})
    order by regDate desc, bno desc
    limit 10;
    ```
- **전체 페이지 수/현재 페이지 번호 개념이 없다.** "총 13페이지 중 3페이지" 같은 표시가 어렵다(필요하면 count는 별도). 무한스크롤은 보통 이런 표시가 필요 없음.

### 7-5. MyBatis 적용 예시

```xml
<select id="getListByCursor" resultType="BoardVO">
  select bno, title, content, writer, regDate, updateDate
  from tbl_board
  <where>
    <if test="lastBno != null">
      bno &lt; #{lastBno}        <!-- XML이라 < 는 &lt; 로 -->
    </if>
  </where>
  order by bno desc
  limit #{amount}
</select>
```
```java
// 첫 페이지: lastBno = null  → where 절 없이 최신 10개
// 다음 페이지: 직전 응답의 마지막 bno를 lastBno로 전달
List<BoardVO> getListByCursor(@Param("lastBno") Long lastBno,
                              @Param("amount") int amount);
```
> XML에서 `<`는 `&lt;`로 써야 한다(태그로 오해받음). `>`는 그냥 써도 됨.

### 7-6. offset vs cursor 최종 비교

| 항목 | offset (`limit n, size`) | cursor (`where bno < ?`) |
|------|--------------------------|---------------------------|
| 임의 페이지 점프 | ✅ 가능 | ❌ 불가(순차만) |
| 깊은 페이지 성능 | ❌ 느려짐 | ✅ 항상 빠름 |
| 전체 페이지 수 표시 | ✅ 쉬움(count) | △ 어려움 |
| 데이터 추가/삭제 중 일관성 | ❌ 중복·누락 가능* | ✅ 안정적 |
| 적합한 UI | 번호 페이징(게시판·관리자) | 무한스크롤·더보기(피드) |

> *offset의 일관성 문제: 1페이지 보는 동안 새 글이 추가되면, 2페이지에서 이미 본 글이 또 보이거나(중복) 어떤 글이 밀려서 안 보일 수(누락) 있다. 커서는 "특정 bno 기준"이라 이런 흔들림이 없다.

### 7-7. 언제 무엇을?
- **게시판/관리자 목록 + 번호 페이징 UI** → offset 방식 (번호 페이징 방식). 인덱스만 있으면 충분.
- **피드/타임라인/모바일 무한스크롤, 데이터 매우 많음** → 커서 방식.

## 8. MyBatis / Spring 적용

### 8-1. 파라미터 객체 (Criteria)
```java
@Data
public class Criteria {
  private int page   = 1;   // 현재 페이지
  private int amount = 10;  // 페이지당 개수

  // LIMIT에 넣을 offset 계산 (표현식 못 넣으니 여기서 값으로)
  public int getOffset() {
    return (page - 1) * amount;
  }
}
```

### 8-2. Mapper
```xml
<!-- 현재 페이지 데이터 -->
<select id="getListWithPaging" resultType="BoardVO">
  select bno, title, content, writer, regDate, updateDate
  from tbl_board
  where bno > 0
  order by bno desc
  limit #{offset}, #{amount}
</select>

<!-- 전체 개수 -->
<select id="getTotalCount" resultType="int">
  select count(*) from tbl_board where bno > 0
</select>
```
> `select *` 보다 **필요한 컬럼만 명시**하는 게 인덱스 활용·성능에 유리.

### 8-3. Mapper 인터페이스
```java
List<BoardVO> getListWithPaging(Criteria cri);
int getTotalCount(Criteria cri);
```
> `cri.getOffset()`, `cri.getAmount()`가 `#{offset}`, `#{amount}`에 자동 매핑된다(getter 기반).

### 8-4. 처리 흐름
```
1. 화면에서 page, amount 받음 → Criteria
2. getListWithPaging(cri)  → 현재 페이지 글 목록
3. getTotalCount(cri)      → 전체 건수
4. 전체 건수로 startPage/endPage/prev/next 계산 (PageDTO)
5. 목록 + 페이지 정보를 JSP로 전달
```

## 9. 실수하기 쉬운 포인트

- **숫자 하나면 size, 둘이면 `offset, size`.** `limit 20`(20개)과 `limit 20, 10`(20개 건너뛰고 10개)은 완전히 다름.
- **order by 빼먹지 말 것.** 안 그러면 페이지마다 순서가 바뀔 수 있음.
- **offset은 0부터.** `(page-1)*amount`이지 `page*amount` 아님.
- **count 쿼리에도 같은 where 조건**을 줘야 함(검색 페이징이면 더 중요). 안 그러면 페이지 수가 안 맞음.
- **페이지 수 계산은 올림(ceil).** 내림하면 마지막 자투리 페이지가 사라짐.
- **LIMIT에 연산식 금지** → 자바에서 계산해서 값으로.

## 10. 한 줄 요약
- **LIMIT**로 자른다 → `limit offset, size`, offset 생략 시 0.
- **offset = (page-1) × amount**, 계산은 자바에서(표현식 못 넣음).
- **order by 필수**(정렬 기준 고정).
- **count(*) 따로 필요** → 전체 페이지 = `ceil(total/amount)`.
- **offset 깊으면 느림** → 인덱스 + (필요시) keyset 페이징.
