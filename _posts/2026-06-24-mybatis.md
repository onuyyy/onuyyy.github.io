## 0. 큰 그림: Mapper XML이 뭐냐

> **"Java 메서드 ↔ 실제 SQL"을 이어주는 연결표.**

Java(Dao)에서
```java
sqlSession.selectList("com.example.common.sql.CodeGroupMapper.searchCodeGroup2", map)
```
라고 부르면 → XML에서 `namespace + id`가 일치하는 SQL을 찾아 실행.

```xml
<mapper namespace="com.example.common.sql.CodeGroupMapper">
   <!-- 이 안의 SQL들을 namespace + id 로 식별 -->
</mapper>
```
- `namespace` = 이 매퍼의 고유 주소. Dao의 `"네임스페이스.id"` 문자열과 **글자 그대로** 일치해야 함.
- 이 예제의 6계층 흐름: `Controller → Service → ServiceImpl → Dao → Mapper.xml → DB`

## 1. SQL 4종 태그 (CRUD)

| 태그 | 용도 | 리턴 |
|------|------|------|
| `<select>` | 조회 | 결과 행 |
| `<insert>` | 입력 | **영향받은 행 수(int)** |
| `<update>` | 수정 | **영향받은 행 수(int)** |
| `<delete>` | 삭제 | **영향받은 행 수(int)** |

- insert/update/delete가 "바뀐 행 수"를 돌려주므로, ServiceImpl에서
  `if(dao.insert(...) == 0) throw new RollbackException();` → **0이면 실패**로 판단.

## 2. 입출력 타입: `parameterType` / `resultType`

```xml
<select id="countCodeGroup2" resultType="int" parameterType="hashMap">
```
- `parameterType` = **넘기는 값**의 타입 (`hashMap`, `string`, `int` …)
- `resultType` = **받는 값**의 타입.

### resultType은 무엇을 기준으로 적나?
> **"한 행(row)이 Java에서 무슨 타입이 되느냐"**

```
SELECT 결과가...
 ├─ 값 하나(컬럼 1개, 행 1개)
 │    ├─ count/개수      → int (또는 long)
 │    ├─ 금액/소수       → BigDecimal   ← double 말고! (오차/금융권 금지)
 │    ├─ 날짜           → string 또는 Date
 │    └─ 문자 한 개      → string
 │
 └─ 컬럼 여러 개(행 전체) → hashMap      ← 이 예제 기본
                          (또는 전용 VO 클래스: resultType="com.xxx.CodeVO")
```

- **행이 여러 개여도 resultType은 안 바뀜.** "여러 행"은 호출 방식(`selectList`)이 처리.
  ```java
  selectOne (...) → HashMap         (한 행)
  selectList(...) → List<HashMap>   (여러 행, resultType은 그대로 "hashMap")
  ```

### resultType은 사실상 `<select>` 전용
| 태그 | parameterType | resultType |
|------|:---:|:---:|
| `<select>` | 필요시 | **필수** |
| `<insert>` | 보통 씀 | ❌ (행 수 자동 반환) |
| `<update>` | 보통 씀 | ❌ |
| `<delete>` | 보통 씀 | ❌ |

- 예외: auto PK 회수 시 `useGeneratedKeys="true" keyProperty="..."` 또는 `<selectKey>`.
  이때도 **메서드 리턴은 여전히 int(행 수)**, 생성된 키는 넘긴 파라미터 map에 꽂혀서 돌아옴.

### resultType에 적을 수 있는 종류 (별칭 alias)

**A. 단일 값 (컬럼 1개)**
```
int / integer / long / short / double / float / boolean
string
decimal / bigdecimal     ← 돈
date                      ← 날짜
object
```

**B. 행 전체 (컬럼 여러 개)**
```
map / hashmap             ← 이 예제 기본. 한 행 = Map
com.패키지.클래스명         ← VO/DTO. 별칭이 아닌 풀 클래스명 그대로
```

**C. 컬렉션 별칭 (잘 안 씀, 헷갈림 주의)**
```
list / arraylist / collection
```
- ⚠️ **"여러 행"은 resultType에 list라고 안 씀.** `selectList()`가 알아서 `List<resultType>`로
  묶음. resultType엔 **항상 "한 행"의 타입**만 적는다.

### "컬럼 여러 개" 받는 3가지 방법 (hashMap이 강제는 아님)

| 방법 | 지정 | 특징 |
|------|------|------|
| ① **Map** | `resultType="hashMap"` | 컬럼명→key 자동. 클래스 불필요. **이 예제 기본** |
| ② **VO/DTO 클래스** | `resultType="com.xxx.CodeGroupVO"` | 필드로 타입 명시. 타입안전·자동완성 ↑ |
| ③ **resultMap** | `resultMap="codeGroupMap"` (resultType 대신) | 컬럼↔필드 수동 매핑. JOIN·중첩객체 |

**① hashMap vs ② VO — 언제 뭘?**

| | hashMap | VO 클래스 |
|--|---------|----------|
| 클래스 생성 | 불필요 (편함) | 필요 (귀찮음) |
| 타입 안전 | ❌ `map.get("XX")` 오타가 런타임에야 발견 | ✅ `vo.getXx()` 컴파일 시 잡힘 |
| 이 예제 | **거의 다 이것** | 일부만 |

```xml
<!-- ② VO -->
<select id="..." resultType="com.example...CodeGroupVO">
```
```java
public class CodeGroupVO {
    private String groupCode;     // GROUP_CODE ↔ groupCode
    private BigDecimal amount;    // 타입을 못박을 수 있음
}
```

**③ resultMap — JOIN/중첩일 때 (지금은 "있다"만 알면 됨)**

`resultType`은 "컬럼명 = 필드명" 자동 매핑. 이름이 안 맞거나 **테이블 JOIN해서 객체 안에 객체**를
넣을 때는 `<resultMap>`을 따로 정의해 수동 매핑:
```xml
<resultMap id="codeGroupMap" type="com.xxx.CodeGroupVO">
  <id     column="GROUP_CODE"    property="groupCode"/>
  <result column="GROUP_CODE_NM" property="groupCodeNm"/>
</resultMap>
<select id="..." resultMap="codeGroupMap">   <!-- resultType 아님! -->
```
- ⚠️ `resultMap="codeGroupMap"`의 `codeGroupMap`은 **컬럼명이 아니라** 내가
  `<resultMap id="codeGroupMap">`으로 정의해둔 "매핑 규칙 블록의 이름". select가 그 이름으로 불러다 씀.
- 이 예제 기본은 hashMap이라 거의 안 만남.

### ⭐ resultType 실 사용 예시 (타입별 XML → Dao → Java)

각 타입을 "XML 정의 → Dao 호출 → Java에서 받기" 흐름으로.

**예시 1) `int` — 개수/존재 여부 (count)**
```xml
<select id="countUser" resultType="int" parameterType="hashMap">
  select count(*) from BR_USER with (nolock)
  <where>
    <if test="DEPT_ID != null"> and DEPT_ID = #{DEPT_ID,jdbcType=VARCHAR} </if>
  </where>
</select>
```
```java
int cnt = sqlSession.selectOne("...UserMapper.countUser", map);
if (cnt > 0) { ... }   // 중복 체크 / 존재 여부에 자주 씀
```

**예시 2) `string` — 단일 값 하나**
```xml
<select id="selectUserName" resultType="string" parameterType="string">
  select USER_NM from BR_USER with (nolock)
   where USER_ID = #{USER_ID,jdbcType=VARCHAR}
</select>
```
```java
String name = sqlSession.selectOne("...UserMapper.selectUserName", "U001");
// "홍길동"
```
- 컬럼 1개·행 1개 → 그 값 하나만 깔끔히 받을 때. (`'Y'`/`'N'` 플래그 조회 등에도 흔함)

**예시 3) `boolean` — Y/N·0/1을 true/false로**
```xml
<select id="hasRole" resultType="boolean" parameterType="hashMap">
  select case when count(*) > 0 then 1 else 0 end
    from BR_USER_ROLE with (nolock)
   where USER_ID = #{USER_ID,jdbcType=VARCHAR}
</select>
```
```java
boolean ok = sqlSession.selectOne("...UserMapper.hasRole", map);
if (!ok) throw new InsufficientRoleException();
```
- DB의 `1/0` 또는 `'Y'/'N'`을 MyBatis가 `boolean`으로 변환. 권한·플래그 판정에 편함.

**예시 4) `hashMap` — 한 행(여러 컬럼)**
```xml
<select id="selectUser" resultType="hashMap" parameterType="string">
  select USER_ID, USER_NM, DEPT_ID, USE_YN
    from BR_USER with (nolock)
   where USER_ID = #{USER_ID,jdbcType=VARCHAR}
</select>
```
```java
HashMap<String,Object> user = sqlSession.selectOne("...UserMapper.selectUser", "U001");
String nm = (String) user.get("USER_NM");   // key = DB 컬럼명 그대로(대문자)
```

**예시 5) `hashMap` + `selectList` — 여러 행**
```xml
<!-- resultType은 똑같이 "hashMap" (한 행의 타입). List는 selectList가 처리 -->
<select id="selectUserList" resultType="hashMap" parameterType="hashMap">
  select USER_ID, USER_NM from BR_USER with (nolock)
  <where>
    <if test="DEPT_ID != null"> and DEPT_ID = #{DEPT_ID,jdbcType=VARCHAR} </if>
  </where>
</select>
```
```java
List<HashMap<String,Object>> list = sqlSession.selectList("...UserMapper.selectUserList", map);
for (HashMap<String,Object> row : list) { row.get("USER_NM"); }
```

**예시 6) VO 클래스 — 타입 안전하게 받기**
```java
public class UserVO {
    private String userId;     // USER_ID
    private String userNm;     // USER_NM
    private boolean useYn;
    // getter/setter ...
}
```
```xml
<select id="selectUserVo" resultType="com.xxx.UserVO" parameterType="string">
  select USER_ID, USER_NM, USE_YN
    from BR_USER with (nolock)
   where USER_ID = #{USER_ID,jdbcType=VARCHAR}
</select>
```
```java
UserVO vo = sqlSession.selectOne("...UserMapper.selectUserVo", "U001");
vo.getUserNm();   // 오타나면 컴파일 에러로 잡힘 ✅ (hashMap의 get("오타")는 못 잡음)
```
> 컬럼명(`USER_NM`) ↔ 필드명(`userNm`)의 언더스코어→카멜 자동매핑은 MyBatis 설정
> (`mapUnderscoreToCamelCase=true`)이 켜져 있어야 함. 안 켜졌으면 이름을 똑같이 맞추거나 resultMap 사용.

> 💡 **참고:** 프로젝트에 따라 `resultType="camelResultMap"` 같은 커스텀 타입을 쓰기도 한다.
> HashMap을 상속한 커스텀 클래스라, 결과 key를 DB 컬럼(`GROUP_CODE_NM`)이 아니라
> **카멜케이스(`groupCodeNm`)로 자동 변환**해줌. `hashMap`(대문자 key)과 `camelResultMap`(카멜 key)
> 두 스타일이 공존할 수 있으니, 새 쿼리 만들 땐 **주변 화면이 어느 쪽을 쓰는지 보고 맞추면** 됨.

## 3. ⭐ 가장 중요: `#{}` vs `${}` (보안 직결, 무조건 암기)

```xml
where GROUP_CODE = #{GROUP_CODE}   -- ✅ 안전 (PreparedStatement, ? 바인딩)
order by ${ORDER_BY}               -- ⚠️ 위험 (문자열을 SQL에 그대로 박음)
```

| | `#{ }` | `${ }` |
|--|--------|--------|
| 처리 | `?` 자리표시자 + 값 바인딩 | **문자열 그대로 치환** |
| SQL Injection | 안전 ✅ | 취약 ❌ |
| 언제 씀 | **거의 다 이것** (값) | 컬럼명/정렬 등 **구조**가 들어갈 때만 |

- `${}`는 불가피할 때만 쓰고 **반드시 화이트리스트 검증**을 동반.
  - 이 예제: `order by ${ORDER_BY}` 전에 Controller에서 `CommonUtil.isOrderby()`로 검증.

## 4. `jdbcType=VARCHAR` — 왜 붙이나? (핵심: null 때문)

```xml
#{USE_YN,jdbcType=VARCHAR}
```
- `jdbcType` = **값을 DB에 넣을 때(input)** 어떤 타입으로 보낼지 명시. (DB 컬럼 타입에 대응)
- **존재 이유의 90% = null 처리.**
  - 값이 있으면 MyBatis가 타입 추측 → 없어도 됨.
  - 값이 **null이면** "이 null을 무슨 타입으로 넣지?"를 못 정해 에러
    (`JdbcType is not set / Invalid column type`). JDBC `setNull()`은 타입을 알아야 작동.
- 방향(direction) 정리:

| 위치 | 방향 | jdbcType |
|------|------|---------|
| `#{ }` (insert/update/where 파라미터) | Java → DB (입력) | **null 대비 명시** ⭐ |
| `resultType` (select 결과) | DB → Java (출력) | 안 씀 (받는 Java 타입만) |

- 결론: 어떤 게 null로 올지 따지기 귀찮으니 **습관적으로 모든 `#{}`에 붙인다.** (안전빵)
- 스타일: 이 예제는 **콤마 뒤 공백 없음** `#{X,jdbcType=VARCHAR}` (공백 있어도 동작은 동일).

## 5. DB 컬럼 타입 ↔ Java 타입 매핑

```
DB 컬럼 타입  ──→  JDBC 타입  ──→  Java 타입
(VARCHAR)        (VARCHAR)       (String)
(DECIMAL)        (NUMERIC)       (BigDecimal)
(DATETIME)       (TIMESTAMP)     (Timestamp)
```
변환은 MyBatis 내부 **TypeHandler**가 표준표대로 자동 수행.

### SQL Server 기준 표준 매핑표
| DB 컬럼 타입 | jdbcType | Java 타입 |
|------|------|------|
| `VARCHAR`, `CHAR`, `NVARCHAR` | `VARCHAR` | **`String`** |
| `INT` | `INTEGER` | **`Integer`** |
| `BIGINT` | `BIGINT` | `Long` |
| `DECIMAL`, `NUMERIC`, `MONEY` | `NUMERIC`/`DECIMAL` | **`BigDecimal`** (돈!) |
| `FLOAT`, `REAL` | `DOUBLE`/`REAL` | `Double`/`Float` |
| `BIT` | `BIT` | `Boolean` |
| `DATETIME`, `DATE` | `TIMESTAMP`/`DATE` | `Timestamp`/`Date` |

- **hashMap 사용 시:** Java 타입을 내가 안 정함 → **DB 컬럼 타입이 Java 타입을 자동 결정**.
  (`map.get("USE_YN")`은 VARCHAR라 자동 String)
- **VO 클래스 사용 시:** 필드에 선언한 Java 타입으로 매핑 → **타입을 직접 통제 가능**.
  - 단, 이 예제는 거의 다 `hashMap`으로 통일 → 일단 hashMap만 써도 됨.

### 💡 실무 관례: 날짜를 일부러 VARCHAR로 저장
```java
requestParam.put("REGIST_DT", DateUtil.getDateString(new Date(), DateUtil.YYYYMMDDHHMMSS));
```
- `REGIST_DT`를 `DATETIME`이 아니라 `VARCHAR`로 두고 `'20260610153000'` 문자열로 저장.
- 이유: `Date/Timestamp`의 타임존·포맷·null 처리가 까다로워 버그가 잦음 → 한국 SI 현장은
  **날짜를 문자열(`yyyyMMddHHmmss`)로 저장**하는 관례가 흔함. Java에서도 String으로 단순 처리.

## 6. 동적 SQL 태그 (MyBatis의 꽃)

조건에 따라 SQL을 **조립**하는 태그들. "파라미터 값에 따라 WHERE 조건이 붙었다 빠졌다"
하는 게 전부 이 태그들 덕분.

### 0) ⭐ `test="..."` 안에 쓰는 건 OGNL (Java 아님!)

`<if>`, `<when>` 등의 `test` 속성 값은 MyBatis가 **OGNL(Object-Graph Navigation Language)**
이라는 표현식 언어로 **런타임에 해석**한다. MyBatis 자체 문법도, 컴파일되는 Java 코드도 아니다.
"Java를 닮은 별도의 작은 언어"라고 보면 된다.

```xml
<if test="USE_YN != null">     <!-- 따옴표 안 "USE_YN != null" 전체가 OGNL 표현식 -->
```

- `test` 안의 `USE_YN` 같은 이름 = 넘긴 **파라미터(map의 key 또는 VO 필드)** 이름.
  → 13장 1번 "map key 대소문자 함정"이 바로 이것. 이름이 글자 그대로 안 맞으면 항상 null.

**Java처럼 보이지만 다른 점 (실전 버그 직결):**

| 하고 싶은 것 | OGNL 권장 | 비고 |
|------|------|------|
| 논리 AND | **`and`** | `&&`도 되지만 XML에선 `&amp;&amp;`로 이스케이프해야 해서 안 씀 |
| 논리 OR | **`or`** | `\|\|`도 동일 이유로 기피 |
| null 체크 | `x != null` | 가장 흔하고 안전 ✅ |
| 숫자/객체 비교 | `==`, `!=`, `>`, `<` | 그대로 사용 가능 |
| 문자열 비교 | `name == "Y"` (**큰**따옴표) | `'Y'`(작은따옴표)는 **char**로 봐서 String과 `==`가 항상 false ⚠️ |
| 빈 문자열까지 | `x != null and x != ''` | null과 `''`는 다름. 빈값도 거르려면 둘 다 |

#### OGNL 연산자 / 문법 전체 치트시트

```
비교    ==  !=  >  <  >=  <=          (키워드형도 됨: eq  neq  gt  lt  gte  lte)
논리    and  or  not                  (&& || ! 도 되나 XML이라 단어형 선호)
산술    +  -  *  /  %
삼항    type == "T" ? 'AND' : 'OR'    (조건 ? 참값 : 거짓값)
포함    status in ('A','B')           /   key not in ('X','Y')
null    x != null,  x == null
빈값    x != null and x != ''         (null과 빈문자 ''는 별개)
프로퍼티 obj.field   → getField() 자동 호출,  깊으면 a.b.c 로 체이닝
인덱스   list[0],  array[0],  map['key']  (또는 map.key)
메서드   obj.method(...)  → 자바 메서드 그대로 호출 (아래 ②)
정적호출 @패키지.클래스@메서드(...) / @클래스@상수   (아래 ②)
리터럴   "문자열"(String) /  'A'(char!) /  123 /  3.14 /  true / false / null
```

#### ① 문자열 비교 — 진짜 함정은 `==` vs `equals`가 아니라 `''`(char) vs `""`(String)

OGNL의 `==`는 내부에서 `.equals()`를 부르므로 **String끼리면 `==`도 정상 동작**한다.
문제는 **따옴표 종류**다.

| OGNL 표현식 | 리터럴의 정체 | 결과 |
|---|---|---|
| `type == 'T'` | `'T'` = **char** | String vs char → **항상 false** ❌ |
| `type == "T"` | `"T"` = **String** | `.equals`로 비교 → **정상** ✅ |
| `type.equals('T')` | `'T'` = char → Character 박싱 | String.equals(Character) → **항상 false** ❌ |
| `type.equals("T")` | `"T"` = **String** | **정상** ✅ |

> ⭐ `==`든 `equals`든 **인자는 반드시 `"..."`(큰따옴표=String)**. `'T'`(작은따옴표)는 char라 무조건 false.

#### ② 메서드 호출도 OGNL이다 (`.equals`는 자바 메서드를 OGNL이 부른 것)

OGNL은 객체의 **자바 메서드를 그대로 호출**할 수 있다.
```xml
<if test='type.equals("T")'>          <!-- String.equals 호출 -->
<if test="list.size() > 0">           <!-- 컬렉션 크기 -->
<if test='name.length() > 0'>         <!-- 문자열 길이 -->
<if test='keyword.indexOf("A") >= 0'> <!-- 포함 검사 -->
<if test="@java.lang.Math@max(a,b) > 10"> <!-- @클래스@static 메서드 -->
```

#### ③ 따옴표 충돌 — 속성은 `'...'`, 문자열 리터럴은 `"..."`

`test="..."`의 바깥이 이미 큰따옴표라, 안에서 또 `"T"`를 쓰면 XML이 깨진다. **바깥/안을 다르게** 쓴다.
```xml
<when test="type.equals("T")">          <!-- ❌ 큰따옴표 안 큰따옴표 → 깨짐 -->
<when test='type.equals("T")'>          <!-- ✅ 속성=작은따옴표, 리터럴=큰따옴표(String) -->
<when test="type.equals(&quot;T&quot;)"> <!-- ✅ 가능하나 지저분 -->
```
> 실무 정석 = **속성 `'...'` + 리터럴 `"..."`** 조합. 이러면 char 함정도 자동 회피.

> 요약: `!= null` 같은 **존재 체크는 안심**, **값 비교(특히 문자열)**할 땐 따옴표 규칙을 의식할 것.

### 1) `<if test="...">` — 조건부 포함
```xml
<if test="USE_YN != null">
  and upper(USE_YN) = #{USE_YN,jdbcType=VARCHAR}
</if>
```
- 가장 기본. `test`가 **참이면** 안쪽 SQL 조각을 붙이고, 거짓이면 통째로 뺀다.
- 위 예: `USE_YN`이 넘어왔을 때만 `and upper(USE_YN) = ?` 조건을 추가.
  안 넘어오면(`null`) 그 줄 자체가 SQL에서 사라져 "전체 조회"가 됨.
- `<if>`는 **else가 없다.** if-else가 필요하면 아래 `<choose>`를 쓴다.

### 2) `<where>` — 똑똑한 WHERE
```xml
<where>
  <if test="LIKE_CODE_GROUP != null"> and upper(GROUP_CODE) like #{LIKE_CODE_GROUP,jdbcType=VARCHAR} </if>
  <if test="USE_YN != null"> and upper(USE_YN) = #{USE_YN,jdbcType=VARCHAR} </if>
</where>
```
- 안쪽 `<if>` 중 **하나라도 참**이면 그 앞에 `WHERE`를 붙이고, **하나도 없으면 `WHERE`를 안 붙인다.**
- 살아남은 첫 조건 맨 앞의 `and`/`or`를 **자동 제거** → `WHERE and ...` 깨짐 방지.
- 그래서 각 `<if>` 안에서 조건마다 앞에 `and`를 붙여도 안전하다(첫 줄 것만 알아서 떼줌).
- 직접 `where 1=1`로 시작해 `and ...`를 잇는 옛날 방식의 깔끔한 대체재.

### 3) `<choose>/<when>/<otherwise>` — switch / if-else
```xml
<choose>
  <when test="ORDER_BY != null"> order by ${ORDER_BY} </when>
  <otherwise> order by REGIST_DT desc </otherwise>
</choose>
```
- 여러 `<when>` 중 **맨 위에서부터 처음 참이 되는 하나만** 실행하고 나머지는 건너뛴다(Java `switch`).
- 어느 `<when>`도 안 맞으면 `<otherwise>`(기본값) 실행. `<otherwise>`는 생략 가능.
- 위 예: 정렬 기준이 넘어오면 그걸로, 없으면 `REGIST_DT desc`로 기본 정렬.
  (`<if>` 여러 개와 달리 **딱 하나만** 선택된다는 게 핵심.)

### 4) `<foreach>` — IN 절 / 반복
```xml
where GROUP_CODE in
<foreach collection="list" item="code" open="(" separator="," close=")">
  #{code}
</foreach>
<!-- 결과: where GROUP_CODE in ('A','B','C') -->
```
- 넘긴 컬렉션을 돌면서 SQL 조각을 **반복 생성**. IN 절 만들 때 제일 많이 씀.
- 속성:
  - `collection` = 돌릴 대상. ⭐ **여기엔 "값"이나 "타입(List)"이 아니라 "이름(key)"을 적는다.**
    `#{USE_YN}`의 `USE_YN`이 이름인 것과 똑같은 원리. → 아래 별도 표 참고.
  - `item` = 반복 중 현재 값을 가리킬 변수명. 이건 반대로 **내가 짓는다**(안에서 `#{code}`로 참조).
  - `open`/`close` = 전체 앞뒤에 한 번씩 붙일 문자(`(` ~ `)`).
  - `separator` = 항목 사이마다 끼울 문자(`,`).

- **⭐ `collection`에 뭘 적나 — "넘기는 방식"이 이름을 결정한다:**

  | 넘기는 방식 (Java) | `collection`에 적는 이름 | 비고 |
  |---|---|---|
  | `List`를 **단독**으로 넘김 | **`list`** | 단독 List에 MyBatis가 붙이는 **자동 기본 이름**(타입명 아님!) |
  | 배열(`String[]`)을 **단독**으로 넘김 | **`array`** | 단독 배열의 자동 기본 이름 |
  | **Map(hashMap)** 에 담아 넘김 | **그 map의 key** | ← **이 예제 기본** |
  | **객체(VO/Criteria)** 에 담아 넘김 | **그 필드 이름** | `getXxx()`로 꺼냄. 배열·List가 필드여도 `array`/`list` 아님! |
  | `@Param("codes")` 로 넘김 | **`codes`** | 어노테이션으로 직접 지정한 이름 |

  > 💡 더 정확히는 `collection` = **"파라미터에서 그 컬렉션까지 가는 경로(이름)"**.
  > 그래서 객체 안에 있으면 필드명, 더 깊으면 `criteria.types`처럼 점(`.`)으로 타고 들어간다.

  ```java
  // ① List 단독 → "list"
  sqlSession.selectList("...Mapper.searchByCodes", Arrays.asList("A","B","C"));
  ```
  ```xml
  <foreach collection="list" item="code" open="(" separator="," close=")"> #{code} </foreach>
  ```
  ```java
  // ② Map에 담아 넘김(이 예제 방식) → map의 key("codeList")
  map.put("codeList", Arrays.asList("A","B","C"));
  sqlSession.selectList("...Mapper.searchByCodes", map);
  ```
  ```xml
  <foreach collection="codeList" item="code" open="(" separator="," close=")"> #{code} </foreach>
  ```
  ```java
  // ③ 객체(VO)에 담아 넘김 → 필드명. 배열이어도 "array"가 아니라 "types"!
  @Data class Criteria { private String[] types; private String keyword; }
  List<BoardVO> getPage(Criteria criteria);   // Criteria 객체 통째로 넘김
  ```
  ```xml
  <!-- criteria.getTypes() 를 돌린다. 같은 이유로 #{keyword}, test="types != null" 도 필드명 그대로 -->
  <foreach collection="types" item="type" open="(" separator="," close=")"> #{type} </foreach>
  ```
  > ⚠️ `"list"`는 "List 타입을 넘긴다"는 뜻이 **아니라** "단독 List의 기본 이름이 마침 `list`"일 뿐.
  > 이 예제는 거의 다 hashMap으로 넘기므로 실전에선 **`collection="내가 put한 key"`** 형태가 더 흔하다.

- ⚠️ 리스트가 **비어 있으면** `in ()`이 되어 SQL 에러 → 보통 `<if test="list != null and list.size > 0">`로 감싼다.

### 5) `<set>` — 똑똑한 UPDATE (`<where>`의 update 버전)
```xml
update BR_CODE_GROUP
<set>
  <if test="GROUP_CODE_NM != null"> GROUP_CODE_NM = #{GROUP_CODE_NM}, </if>
  <if test="USE_YN != null"> USE_YN = #{USE_YN}, </if>
</set>
where GROUP_CODE = #{GROUP_CODE}
<!-- 끝의 불필요한 콤마(,)를 자동 제거 -->
```
- UPDATE의 `SET` 절을 동적으로 조립. **넘어온 컬럼만 골라서** 수정(부분 수정).
- 살아남은 마지막 조각 끝의 **불필요한 콤마(`,`)를 자동 제거** → `USE_YN = ?,  where` 깨짐 방지.
- ⚠️ 모든 `<if>`가 거짓이면 `SET`이 비어 `update ... where ...`가 되어 SQL 에러.
  "수정할 값이 최소 1개는 있다"가 보장될 때 안전.

### 특수 변수 `_parameter`
```xml
<if test="_parameter != null">
```
- `_parameter` = 넘어온 **파라미터 객체 전체**를 가리키는 MyBatis 예약어. "파라미터 자체가 null 아니면".
- (참고 예약어) `_databaseId` = `databaseIdProvider` 설정 시 현재 DB 종류 식별자.

### 페이징 (MSSQL)
```xml
<if test="startPageIdx != null and endPageIdx != null">
  offset #{startPageIdx,jdbcType=INTEGER} rows fetch next #{endPageIdx,jdbcType=INTEGER} rows only
</if>
```
- 동적 태그 자체는 아니고, 위 태그들을 **실제로 조합한 페이징 패턴**.
- 시작·끝 인덱스가 **둘 다** 넘어왔을 때만(`and`) SQL Server의 `OFFSET ... FETCH NEXT` 절을 붙임.
- `offset N rows` = 앞 N건 건너뜀, `fetch next M rows only` = 그다음 M건만. (MSSQL 2012+ 표준 페이징)

### 6) 재사용 — `<sql>` + `<include>`
SQL 조각을 한 번 정의해 여러 쿼리에서 끼워 쓴다. (동적 태그는 아니지만 단짝)
```xml
<!-- ① 공통 조각 정의 (id로 식별) -->
<sql id="searchWhere">
  <where>
    <if test="LIKE_CODE_GROUP != null"> and upper(GROUP_CODE) like #{LIKE_CODE_GROUP,jdbcType=VARCHAR} </if>
    <if test="USE_YN != null"> and upper(USE_YN) = #{USE_YN,jdbcType=VARCHAR} </if>
  </where>
</sql>

<!-- ② list·count가 같은 조각을 불러다 씀 -->
<select id="searchCodeGroup" resultType="hashMap" parameterType="hashMap">
  select GROUP_CODE, GROUP_CODE_NM from BR_CODE_GROUP with (nolock)
  <include refid="searchWhere"/>
  order by REGIST_DT desc
</select>
<select id="countCodeGroup" resultType="int" parameterType="hashMap">
  select count(*) from BR_CODE_GROUP with (nolock)
  <include refid="searchWhere"/>
</select>
```
- `<sql>`은 `id`, `<include>`는 `refid`로 가리킴(글자 그대로 일치). 다른 매퍼 건 `refid="namespace.id"`.
- ⭐ **최고 효용 = list/count 검색조건 통일** → 13장 2번 함정("건수 3인데 목록 10개") 원천 차단.
- 값 주입: `<include><property name="alias" value="g"/></include>` → 조각 안 `${alias}`로 받음.
  단 `${}`라 **고정 상수만**(사용자 입력 금지).

### 7) `<bind>` — OGNL로 변수 만들기 (LIKE 패턴에 자주)
```xml
<bind name="kw" value="'%' + keyword + '%'"/>
... writer like #{kw}
```
- `value`는 **OGNL** → `+`로 문자열 결합. 결과를 `#{kw}`로 **안전하게 바인딩**(8장 LIKE 참고).

### 8) `<trim>` — `<where>`/`<set>`의 일반형 (직접 제어)
```xml
<trim prefix="WHERE" prefixOverrides="AND |OR ">
  <if test="USE_YN != null"> and USE_YN = #{USE_YN} </if>
</trim>
```
- `prefix` = 안에 내용이 있으면 맨 앞에 붙일 글자 / `suffix` = 맨 뒤에 붙일 글자.
- `prefixOverrides` = 앞에서 지울 후보(`|`로 나열) / `suffixOverrides` = 뒤에서 지울 후보.
- `<where>` ≒ `<trim prefix="WHERE" prefixOverrides="AND |OR ">`, `<set>` ≒ `<trim prefix="SET" suffixOverrides=",">`.
  → 평소엔 `<where>`/`<set>`로 충분, 특이 케이스만 `<trim>`.

### 9) batch insert — `<foreach>`로 여러 행 한 번에
```xml
<insert id="insertCodes" parameterType="java.util.List">
  insert into BR_CODE (CODE, CODE_NM) values
  <foreach collection="list" item="r" separator=",">
    (#{r.code,jdbcType=VARCHAR}, #{r.codeNm,jdbcType=VARCHAR})
  </foreach>
</insert>
<!-- 결과: insert into BR_CODE (CODE, CODE_NM) values (?,?),(?,?),(?,?) -->
```
- `separator=","`로 `(...),(...)` 묶음 생성. `item`의 필드는 `#{r.code}`처럼 점으로 접근.
- ⚠️ 한 번에 너무 많으면 파라미터 수 한계/패킷 초과 → 보통 500~1000행씩 끊어서.

### 빈 문자열까지 거르기
```xml
<if test="USE_YN != null and USE_YN != ''">   <!-- null도 빈값('')도 아닐 때만 -->
```
- 화면에서 빈 문자열로 넘어오는 경우가 많아, 검색조건엔 `!= null` 만으로 부족할 때가 잦다.

## 7. CRUD 실전 예제 (INSERT / UPDATE / DELETE)

> 지금까지 예제는 SELECT 위주였으니, 나머지 3종과 PK 회수까지.

### INSERT
```xml
<insert id="insertCode" parameterType="hashMap">
  insert into BR_CODE (CODE, CODE_NM, USE_YN, REGIST_DT)
  values (#{CODE,jdbcType=VARCHAR}, #{CODE_NM,jdbcType=VARCHAR},
          #{USE_YN,jdbcType=VARCHAR}, #{REGIST_DT,jdbcType=VARCHAR})
</insert>
```
```java
int n = sqlSession.insert("...CodeMapper.insertCode", map);   // 리턴 = 들어간 행 수
if (n == 0) throw new RollbackException();                     // 0이면 실패 처리
```

### UPDATE (부분 수정은 `<set>` 활용 — 6장 5번)
```xml
<update id="updateCode" parameterType="hashMap">
  update BR_CODE
  <set>
    <if test="CODE_NM != null"> CODE_NM = #{CODE_NM,jdbcType=VARCHAR}, </if>
    <if test="USE_YN != null"> USE_YN = #{USE_YN,jdbcType=VARCHAR}, </if>
  </set>
  where CODE = #{CODE,jdbcType=VARCHAR}
</update>
```

### DELETE
```xml
<delete id="deleteCode" parameterType="string">
  delete from BR_CODE where CODE = #{CODE,jdbcType=VARCHAR}
</delete>
```
- insert/update/delete 셋 다 **리턴은 영향받은 행 수(int)**. resultType 안 씀(1장).

### auto PK 회수 — `<selectKey>` / `useGeneratedKeys`
INSERT 후 **자동 생성된 키 값**을 다시 받아와야 할 때.

**방법 A) `useGeneratedKeys` (IDENTITY/AUTO_INCREMENT 컬럼)**
```xml
<insert id="insertBoard" parameterType="hashMap"
        useGeneratedKeys="true" keyProperty="BNO" keyColumn="BNO">
  insert into BR_BOARD (TITLE, CONTENT) values (#{TITLE}, #{CONTENT})
</insert>
```
**방법 B) `<selectKey>` (시퀀스/직접 채번. order=BEFORE는 insert 전 실행)**
```xml
<insert id="insertBoard" parameterType="hashMap">
  <selectKey keyProperty="BNO" resultType="int" order="BEFORE">
    select next value for SEQ_BOARD
  </selectKey>
  insert into BR_BOARD (BNO, TITLE) values (#{BNO}, #{TITLE})
</insert>
```
```java
int n = sqlSession.insert("...BoardMapper.insertBoard", map);
Object bno = map.get("BNO");   // ⭐ 메서드 리턴은 여전히 행 수(int).
                               //    생성된 키는 넘긴 map(파라미터)에 꽂혀서 돌아온다
```
- `keyProperty` = 채운 키를 **파라미터의 어느 필드/키에 담을지**. (위에선 map의 "BNO")
- MSSQL IDENTITY는 A, 시퀀스는 B. `order` = `BEFORE`(insert 전) / `AFTER`(insert 후, IDENTITY류).

## 8. LIKE 검색 패턴 (`#{}`는 따옴표 안에서 치환 안 됨!)

`like '%#{keyword}%'`는 **안 된다.** `#{}`는 `?`로 바뀌는데 따옴표 안에 갇히면 `'%?%'`라는
**글자**가 돼버려 바인딩이 안 되기 때문.

| 방식 | 됨? | 예 |
|---|---|---|
| `like '%#{keyword}%'` | ❌ | `?`가 따옴표 안에 갇힘 |
| DB 함수 결합 | ✅ | MSSQL: `like '%' + #{keyword} + '%'` / 2012+: `like concat('%',#{keyword},'%')` |
| Java에서 `%` 포함 | ✅ | `map.put("keyword","%"+kw+"%")` → `like #{keyword}` (제일 깔끔) |
| `<bind>` | ✅ | `<bind name="kw" value="'%'+keyword+'%'"/>` → `like #{kw}` |
| `like '%${keyword}%'` | ⚠️ | 동작하나 **SQL Injection 취약** → 검색어엔 금지 |

```xml
<!-- MSSQL 권장 (전통 결합 +) -->
where writer like '%' + #{keyword,jdbcType=VARCHAR} + '%'
```
> 이 예제는 MSSQL이라 `+` 결합이 흔함. `concat`은 MySQL 교본 예제 스타일.

## 9. ⚠️ CDATA / XML 이스케이프 (`<`, `>`, `&`)

Mapper는 **XML**이라 SQL에 `<`, `>`, `&`를 그냥 쓰면 **XML 파싱이 깨진다.** (`<`를 태그 시작으로 오해)
```xml
<!-- ❌ 깨짐: 파서가 < 를 태그로 봄 -->
where REGIST_DT &lt; #{END_DT} and CNT > 0
```
해결 2가지:

**① 엔티티 이스케이프** — `<` → `&lt;`, `>` → `&gt;`, `&` → `&amp;`
```xml
where REGIST_DT &lt;= #{END_DT} and AMOUNT &gt; 0
```
**② `<![CDATA[ ... ]]>`** — 이 블록 안은 XML이 해석 안 함(그대로 SQL) ← 부등호 많을 때 편함
```xml
<![CDATA[
  where REGIST_DT >= #{START_DT}
    and REGIST_DT <= #{END_DT}
    and AMOUNT <> 0
]]>
```
- ⚠️ **CDATA 안에서는 `<if>` 같은 동적 태그가 동작 안 한다**(전부 글자로 취급).
  동적 조건과 부등호가 섞이면 → CDATA로 **부등호 부분만 감싸거나**, `&lt;`로 쓴다.
```xml
<if test="END_DT != null"> and REGIST_DT <![CDATA[ <= ]]> #{END_DT} </if>
```

## 10. typeAlias — `resultType="BoardVO"`가 풀패키지 없이 되는 이유

원래는 `resultType="com.example...BoardVO"`처럼 **풀 클래스명**을 써야 하지만,
미리 **별칭(alias)**을 등록해두면 짧은 이름만으로 쓸 수 있다.
```xml
<!-- mybatis-config.xml -->
<typeAliases>
  <typeAlias type="com.example.board.vo.BoardVO" alias="BoardVO"/>
  <!-- 또는 패키지 통째로: 클래스명(첫 글자 소문자 무관)이 자동 별칭 -->
  <package name="com.example.board.vo"/>
</typeAliases>
```
- `int`, `string`, `hashMap`, `list` 같은 것도 **MyBatis 기본 내장 별칭**이다(2장 표).
- `camelResultMap` 같은 커스텀 타입도 이렇게 등록된 별칭(2장 팁).
- 별칭이 없으면 → **풀 클래스명**으로 써야 한다.

## 11. 다중 파라미터 / `@Param` (map 안 쓰고 인자 여러 개)

이 예제는 보통 hashMap 하나로 넘기지만(0장), Mapper 인터페이스 방식에선 인자를 여러 개 넘긴다.
```java
List<BoardVO> search(@Param("type") String type, @Param("keyword") String keyword);
```
```xml
<select id="search" resultType="BoardVO">
  select * from BR_BOARD
  <where>
    <if test="type != null"> and TYPE = #{type} </if>       <!-- @Param 이름으로 참조 -->
    <if test="keyword != null"> and TITLE like '%' + #{keyword} + '%' </if>
  </where>
</select>
```
- `@Param("type")`의 이름 = XML의 `#{type}` / `test="type"`에서 쓰는 이름.
- `@Param` **안 붙이고** 인자 여러 개면 → `#{param1}`, `#{param2}`(순서) 또는 `#{arg0}`로만 접근(가독성 ↓).
- 인자가 **1개**면 이름 없이 `#{아무이름}`이라도 그 값으로 매핑(단독 파라미터).

## 12. resultMap 심화 — JOIN 결과를 객체 안 객체로 (`<association>`/`<collection>`)

2장에서 "있다"만 본 resultMap의 실제 용도. **JOIN 결과**를 중첩 객체로 매핑.

**`<association>` — 1:1 (객체 안에 객체 하나)**
```xml
<resultMap id="boardMap" type="BoardVO">
  <id     column="BNO"   property="bno"/>
  <result column="TITLE" property="title"/>
  <association property="writer" javaType="UserVO">   <!-- BoardVO.writer (UserVO 1개) -->
    <id     column="USER_ID" property="userId"/>
    <result column="USER_NM" property="userNm"/>
  </association>
</resultMap>
```
**`<collection>` — 1:N (객체 안에 리스트)**
```xml
<resultMap id="boardMap" type="BoardVO">
  <id     column="BNO" property="bno"/>
  <collection property="replies" ofType="ReplyVO">    <!-- BoardVO.replies (List<ReplyVO>) -->
    <id     column="RNO"     property="rno"/>
    <result column="REPLY"   property="reply"/>
  </collection>
</resultMap>
```
- `<id>` = PK 컬럼 매핑(중복 행을 한 객체로 묶는 기준). `<result>` = 일반 컬럼.
- `javaType`(1:1) / `ofType`(1:N 안의 원소 타입) 지정에 주의.
- 자동매핑: 컬럼명=필드명이면 생략 가능. 언더스코어→카멜은 `mapUnderscoreToCamelCase=true` 설정 필요(5장).
- 이 예제는 hashMap 기본이라 자주 안 쓰지만, **JOIN을 객체로 받을 땐 이게 정석**.

## 13. ⚠️ 실전에서 자주 터지는 함정 (손코딩 체크리스트)

1. **map key 대소문자 구분** — Controller `searchMap.put("USE_YN", ..)`의 key와
   XML `<if test="USE_YN">` / `#{USE_YN}`의 이름이 **글자 그대로** 같아야 함.
   `useYn`로 담고 `USE_YN`로 꺼내면 → 항상 null → 조건이 영원히 안 걸림. (가장 흔한 버그 1위)
2. **count 쿼리와 list 쿼리의 검색조건은 항상 동일하게** — 한쪽만 필터하면
   "건수는 3인데 목록은 10개" 페이징 깨짐 발생.
3. **계층을 만들면 끝까지 다 연결** — XML에 SQL 만들었으면 Dao·Service·Impl·Controller까지
   호출선이 빠짐없이 이어졌는지 추적. "만들었는데 아무도 안 부르는 죽은 코드" 주의.
4. **count 쿼리엔 order by / 페이징(offset) 넣지 말 것** — 총 건수만 세면 됨.
5. **돈/비율은 BigDecimal, double 금지.**

## 14. 한 장 요약 (책상에 붙이기)

```
<select|insert|update|delete id=".." parameterType=".." resultType="..">
  #{값}            → 안전(PreparedStatement). 거의 이것만 써라
  ${구조}          → 컬럼/정렬만. 반드시 화이트리스트 검증 동반
  ,jdbcType=..     → 값을 DB에 넣을 때 null 대비. 습관적으로 붙여라
  resultType       → select 전용. 값 하나면 그 타입, 여러 컬럼이면 hashMap
                     (insert/update/delete는 행 수 int 자동반환, 안 적음)
  <if test="x != null"> 조건부 </if>   ← test 안은 OGNL ('A'=char, "A"=String)
  <where>          → WHERE + 앞 and/or 자동정리
  <set>            → UPDATE의 콤마 자동정리
  <trim>           → where/set의 일반형 (prefix/suffix/Overrides)
  <choose><when><otherwise> → if-else
  <foreach>        → IN절/반복/batch insert (collection=이름, item=내가짓는변수)
  <sql>+<include>  → SQL 조각 재사용 (list/count 조건 통일)
  <bind>           → OGNL 변수 (LIKE 패턴 등)
  <selectKey>/useGeneratedKeys → auto PK 회수 (리턴은 여전히 행 수)
  like             → '%'+#{kw}+'%'  ('%#{kw}%'는 안 됨!)
  <![CDATA[ < > ]]> 또는 &lt; &gt;  → SQL 부등호 XML 깨짐 방지
</select>
```
