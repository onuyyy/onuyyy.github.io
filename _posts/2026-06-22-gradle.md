> 빌드 도구 Gradle을 "왜 이렇게 동작하나"부터 실무 명령어까지 정리.
> 예시는 WAR + Gretty 기반 레거시 프로젝트(`build.gradle`, Gradle 7.6.4 wrapper, Java 8) 기준.

---

## 0. Gradle이 뭐고 왜 쓰나

- **빌드 자동화 도구**. 소스 컴파일 → 의존성(라이브러리) 다운로드 → 테스트 → 패키징(WAR/JAR) → 실행까지 자동화.
- 비슷한 것: Maven(`pom.xml`, XML 기반). Gradle은 **Groovy/Kotlin 스크립트** 기반이라 더 유연하고 빠름(증분 빌드·캐시).
- Maven(`pom.xml`)에서 Gradle로 이전한 프로젝트라면 `build.gradle` 주석에 "pom.xml 대응" 흔적이 남기도 한다.

---

## 1. 핵심 개념 5가지 (이것만 잡으면 됨)

### ① Task (태스크) = "실행 단위"
Gradle이 하는 모든 일은 **task**. `build`, `war`, `clean`, `appRun` 전부 task.
```cmd
gradlew.bat war        REM war 라는 task 실행
gradlew.bat tasks      REM 실행 가능한 task 목록 보기
```

### ② Plugin (플러그인) = "task 묶음을 추가"
Gradle 코어엔 task가 거의 없다. **플러그인을 적용하면 그 플러그인이 task를 끼워넣는다.**
예시(`build.gradle` 상단):
```groovy
plugins {
    id 'java'          // → compileJava, test, jar, classes ...
    id 'war'           // → war (WAR 패키징)
    id 'eclipse'       // → eclipse (.classpath/.project 생성)
    id 'org.gretty' version '3.1.2'   // → appRun, appStart, appStop ... ★
}
```
| task | 어디서 옴 |
|---|---|
| `appRun`/`appStart`/`appStop` | **Gretty 플러그인** (내장 톰캣 실행) |
| `war` | war 플러그인 |
| `build`/`clean`/`test`/`jar` | java 플러그인 |
| `dependencies`/`tasks`/`wrapper` | Gradle 코어 (플러그인 없어도 있음) |
| `bootRun` | Spring Boot 플러그인 → **부트가 아니면 없음** |

> 그래서 "무슨 명령 쓰지?" 모르면 항상 `gradlew tasks` 로 확인. 프로젝트마다 다름.

### ③ Dependencies (의존성) = "쓰는 라이브러리"
```groovy
dependencies {
    implementation "org.springframework:spring-webmvc:5.3.39"   // 그룹:이름:버전
    testImplementation "..."     // 테스트에서만
    providedCompile "..."        // 컴파일엔 필요, 패키징엔 제외(WAR에서 톰캣 제공 라이브러리)
}
```
**configuration(범위)** 종류:
| 범위 | 의미 |
|---|---|
| `implementation` | 컴파일+런타임 사용 (가장 많이 씀) |
| `testImplementation` | 테스트에서만 |
| `providedCompile` / `compileOnly` | 컴파일만, 패키징 제외 (서버가 주는 것) |
| `runtimeOnly` | 런타임만 (예: JDBC 드라이버) |

### ④ Repository (저장소) = "라이브러리 받아오는 곳"
```groovy
repositories {
    mavenCentral()                    // 공개 중앙 저장소 (인터넷)
    maven { url file('reference/repo') }   // ★ 프로젝트 내 로컬 저장소 (사내 JAR)
}
```
> 폐쇄망(오프라인) 환경에서는 사내 전용 JAR을 `reference/repo` 같은 프로젝트 내 로컬 저장소에서 받기도 한다.

### ⑤ Lifecycle (생명주기) = "task는 의존 순서로 줄줄이 실행"
task는 서로 의존한다. 상위 task를 부르면 하위가 먼저 실행됨:
```
build → war → classes → compileJava   (compileJava부터 거꾸로 실행)
```
```cmd
gradlew.bat build    REM compileJava→test→war→build 까지 다 실행
gradlew.bat war      REM compileJava→classes→war 까지만
```

## 2. 프로젝트의 Gradle 파일들 — 뭐가 뭔지

크게 **빌드 설정 파일** 과 **래퍼(wrapper) 파일** 두 묶음.

```
프로젝트 루트/
├─ settings.gradle        ┐
├─ build.gradle           ├─ 빌드 설정 (내가 작성/관리)
├─ gradle.properties      ┘
├─ gradlew                ┐
├─ gradlew.bat            ├─ 래퍼 (Gradle 자동 설치·실행)
└─ gradle/
   └─ wrapper/
      ├─ gradle-wrapper.jar         ┘
      └─ gradle-wrapper.properties
```

### 2.1 빌드 설정 파일

| 파일 | 역할 | 핵심 |
|---|---|---|
| `settings.gradle` | **빌드 구성** — Gradle이 **가장 먼저** 읽음 | 루트 프로젝트명(`rootProject.name`), 멀티모듈 등록(`include`). "여기가 빌드 루트다" 표시 |
| `build.gradle` | **빌드 내용** — 빌드 로직 본체 | 플러그인·의존성·저장소·태스크 (§1, §6) |
| `gradle.properties` | **빌드 환경설정값** | `org.gradle.java.home`(빌드용 JDK), `jvmargs`(힙), `file.encoding`. PC마다 다를 수 있음 |

> `settings.gradle` = **"빌드의 목차/구성"**, `build.gradle` = **"실제 빌드 내용"**.
> 단일 프로젝트면 `settings.gradle`은 이름만 있거나 거의 비어있기도 함.

### 2.2 Wrapper(래퍼) — "Gradle 설치 안 해도 되게 해주는 장치" ★

**목적**: PC에 Gradle을 따로 설치하지 않아도, **프로젝트에 지정된 버전의 Gradle을 자동으로 받아 실행**.

| 파일 | 역할 | 내가 건드리나 |
|---|---|---|
| `gradlew` | 래퍼 실행 **스크립트 (Linux/Mac)** | ❌ 자동생성 |
| `gradlew.bat` | 래퍼 실행 **스크립트 (Windows)** | ❌ |
| `gradle/wrapper/gradle-wrapper.jar` | 래퍼의 **두뇌** — Gradle 다운로드·압축해제·실행 로직(자바) | ❌ |
| `gradle/wrapper/gradle-wrapper.properties` | **쓸 Gradle 버전 + 다운로드 URL** | 버전 바꿀 때만 |

- `gradlew`/`gradlew.bat`는 **같은 일, OS만 다름**. 스크립트는 짧은 껍데기이고 **진짜 일은 `gradle-wrapper.jar`**가 함.
- `gradle-wrapper.properties` 핵심:
  ```properties
  distributionUrl=https\://services.gradle.org/distributions/gradle-7.6.4-bin.zip
  #              └ 어떤 버전을, 어디서 받을지   (예: 7.6.4)
  distributionPath=wrapper/dists
  #              └ 받은 걸 어디에 풀지 (~/.gradle/wrapper/dists)
  ```
  > 이 파일의 버전만 바꾸면 프로젝트가 쓰는 Gradle 버전이 바뀜.
  > **폐쇄망 이슈**: `distributionUrl`이 인터넷 주소라 그 zip을 못 받으면 래퍼가 멈춤 → 미리 받아두거나 사내 URL로 교체.

- 장점: 누가 빌드해도 **같은 Gradle 버전** 보장 (CI도 wrapper 사용).
- ⚠️ **래퍼 4종(`gradlew`,`gradlew.bat`,`gradle-wrapper.jar`,`gradle-wrapper.properties`)은 절대 지우지 말 것.** 없으면 `./gradlew` 안 돌아감. git에 항상 커밋돼 있어야 정상.

### 2.3 실행 흐름 한 장
```
./gradlew war  (또는 gradlew.bat war)
   1. gradlew 스크립트 시작
   2. gradle-wrapper.jar 실행 (두뇌)
   3. gradle-wrapper.properties 읽음 → "gradle 7.6.4 필요하네"
   4. 해당 버전 있나 확인(~/.gradle/wrapper/dists)
        없으면 distributionUrl에서 다운로드   ← 폐쇄망 막히는 지점
   5. 그 Gradle로 settings.gradle → build.gradle 읽고
   6. war 태스크 실행
```

## 3. 명령어 모음 (실무)

### 3.1 빌드/정리
```cmd
gradlew.bat clean                  REM build 폴더 삭제
gradlew.bat war                    REM WAR 생성 → build\libs\myapp.war
gradlew.bat clean war -x test      REM ★ 자주 쓰는 표준 (CI/Docker 동일)
gradlew.bat build                  REM 컴파일+테스트+패키징 전체
gradlew.bat assemble               REM 테스트 없이 패키징만
```
> `-x test` = test task 제외(eXclude). `-x <task>` 로 아무 task나 건너뛸 수 있음.

### 3.2 실행 (Gretty)
```cmd
gradlew.bat appRun       REM ★ 내장 톰캣8.5 실행, http://localhost:8080 (콘솔 점유)
gradlew.bat appStart     REM 백그라운드 시작
gradlew.bat appStop      REM 정지
gradlew.bat appRestart   REM 재시작
```

### 3.3 진단/정보
```cmd
gradlew.bat tasks                  REM task 목록 (그룹별)
gradlew.bat tasks --all            REM 숨은 task까지 전부
gradlew.bat dependencies           REM 의존성 트리 (충돌 추적)
gradlew.bat dependencies --configuration runtimeClasspath
gradlew.bat -v                     REM Gradle/JVM/OS 버전
gradlew.bat properties             REM 프로젝트 속성
gradlew.bat help --task war        REM 특정 task 설명
```

### 3.4 자주 붙이는 옵션 플래그
```cmd
--offline               REM ★ 폐쇄망: 네트워크 안 쓰고 캐시로만
--refresh-dependencies  REM 의존성 강제 재다운로드
--stacktrace            REM 에러 스택트레이스 (원인 추적)
-i  (--info)            REM 자세한 로그
-d  (--debug)           REM 더 자세히
--no-daemon             REM 데몬 안 쓰고 1회성 (CI에서)
--no-build-cache        REM 캐시 무시하고 새로
-x <task>               REM 특정 task 제외
```

### 3.5 조합 예시
```cmd
gradlew.bat clean war -x test --offline           REM 폐쇄망 빌드
gradlew.bat clean war -x test --stacktrace -i      REM 빌드 깨질 때 원인 추적
gradlew.bat dependencies > deps.txt                REM 의존성 트리 파일로
```

## 4. Daemon / Cache — "왜 두 번째 빌드는 빠른가"

### Daemon (데몬)
- Gradle은 **백그라운드 프로세스(데몬)** 를 띄워 JVM을 재활용 → 두 번째부터 빠름.
- 문제 생기면: `gradlew.bat --stop` (데몬 종료 후 재시도). CI는 `--no-daemon`.

### Cache (캐시) — 폐쇄망에 중요 ★
| 위치 | 내용 |
|---|---|
| `C:\Users\<나>\.gradle\caches` | 다운받은 의존성 라이브러리 |
| `C:\Users\<나>\.gradle\wrapper\dists` | Gradle 배포본(7.6.4) |
| `<프로젝트>\build` | 빌드 산출물(컴파일 클래스, WAR) |
| `<프로젝트>\.gradle` | 프로젝트별 증분빌드 정보 |
> 폐쇄망 반입 = 외부망에서 한 번 빌드 후 위 `.gradle` 폴더들을 통째로 복사 → `--offline` 가능.


## 6. build.gradle 구조 한눈 (예시)

```groovy
plugins { ... }                    // ① 플러그인 (java, war, gretty)
group = 'com.example.demo'         // ② 프로젝트 좌표
version = 'v4.0.0-20251210'
java { sourceCompatibility = 1.8 } // ③ 자바 버전 (★ 8 고정)
sourceSets { ... }                 // ④ 소스 경로 (webapp을 resources로)
ext { springVersion = '5.3.39' }   // ⑤ 버전 변수 (재사용)
repositories { ... }               // ⑥ 저장소 (mavenCentral + reference/repo)
dependencies { ... }               // ⑦ 의존성 목록
war { ... }                        // ⑧ WAR 설정
gretty { httpPort=8080; ... }      // ⑨ 실행 설정 (내장 톰캣)
```

## 7. 자주 만나는 문제

| 증상 | 원인 / 해결 |
|---|---|
| `Could not target platform: Java SE 8` | JDK가 8 아님 → `gradle.properties`의 `org.gradle.java.home` |
| `Could not resolve <라이브러리>` | 저장소에 없음 → 폐쇄망 캐시 반입 또는 `--refresh-dependencies` |
| `Task 'bootRun' not found` | 부트 플러그인 없음 → Gretty면 `appRun` 사용 |
| 빌드가 멈춰있음(hang) | 데몬 이슈 → `gradlew --stop` 후 재시도 |
| 이상하게 옛날 결과 | 캐시 → `clean` 후 빌드, 또는 `--no-build-cache` |
| 권한 거부(`gradlew: Permission denied`) | Linux/Mac → `chmod +x gradlew` |

## 8. 꼭 기억할 것 3가지
1. **항상 wrapper** (`gradlew.bat`), 시스템 Gradle 설치 불필요.
2. **task는 플러그인이 준다** → 모르면 `gradlew tasks`.
3. **자주 쓰는 3대 명령**:
   ```cmd
   gradlew.bat clean war -x test    REM 빌드
   gradlew.bat appRun               REM 실행
   gradlew.bat dependencies         REM 의존성 확인
   ```

## 부록: Java 명령어 빠른 참고
```cmd
java -version / javac -version          REM 버전 확인 (예: 1.8)
java -jar app.jar                       REM jar 실행 (부트 방식)
java -Dspring.profiles.active=local -Xmx2048m -jar app.jar
jar -tf app.war                         REM war 내용 보기
jps                                     REM 자바 프로세스 목록(PID)
jstack <PID>                            REM 스레드 덤프(행/데드락)
keytool -list -keystore x.jks           REM 인증서/키스토어 확인
```
> WAR 프로젝트는 배포 시 `java -jar`가 아니라 **WAR를 Docker 톰캣 ROOT.war로** 올린다.
> 부트의 `bootRun` ↔ Gretty `appRun`, 부트의 `java -jar` ↔ WAR 배포.
