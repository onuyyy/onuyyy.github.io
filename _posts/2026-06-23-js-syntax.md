## 1. 변수 — `var` / `let` / `const`

| 키워드 | 재할당 | 스코프 | 호이스팅 | 권장도 |
|--------|:------:|--------|----------|:------:|
| `var` | O | 함수 스코프 | 됨(`undefined`로 초기화) | ✕ 옛날 |
| `let` | O | 블록 스코프 `{}` | TDZ¹ | △ 바뀌는 값 |
| `const` | ✕ | 블록 스코프 `{}` | TDZ¹ | ◎ 기본값 |

¹ TDZ(Temporal Dead Zone): 선언 전에 쓰면 `ReferenceError`. `var`처럼 `undefined`로 슬쩍 넘어가지 않아 더 안전하다.

```javascript
const bno = '${vo.bno}'                       // 안 바뀜 → const
const actionForm = document.querySelector("#actionForm")
let count = 0                                  // 바뀔 값만 let
```

> **주의 — `const`는 "재할당 금지"지 "불변"이 아니다.** 객체·배열의 내부는 바꿀 수 있다.
> ```javascript
> const cri = { pageNum: 1 }
> cri.pageNum = 2      // ✅ OK (내부 변경)
> cri = {}             // ❌ TypeError (재할당)
> ```

**원칙: 일단 전부 `const`, 재할당이 꼭 필요할 때만 `let`, `var`는 안 쓴다.**

## 2. 템플릿 리터럴 `` `${}` `` — 그리고 JSP EL과의 충돌 ⚠️

백틱(`` ` ``) 문자열 안에서 `${표현식}`으로 값을 끼워 넣는다.

```javascript
const bno = 10
const url = `/board/modify/${bno}`   // "/board/modify/10"
```

> ⚠️ **보간은 백틱 안에서만 된다.** 따옴표(`'`, `"`)로 감싸면 `${}`는 그냥 글자로 출력된다.
> ```javascript
> const test = 0
> `값 ${test}`   // "값 0"        ← 백틱 ✅ 보간됨
> "값 ${test}"   // "값 ${test}"  ← 큰따옴표 ❌ 글자 그대로
> '값 ${test}'   // "값 ${test}"  ← 작은따옴표 ❌ 글자 그대로
> ```
> 외운다면 한 줄: **`\`(백슬래시)는 "JSP야 비켜" 신호, 백틱은 "JS야 변수 채워" 신호.**

### 문제: JSP EL `${}` 와 모양이 똑같다

JSP에서 `${...}`는 **서버가 처리하는 EL**이고, JS 템플릿 리터럴 `${...}`는 **브라우저가 처리**한다.
둘이 생김새가 같아서 `<script>` 안에서 섞이면 충돌한다.

**누가 언제 처리하나 (순서가 핵심):**

```
JSP 파일
  │  ① 서버: ${...} 발견 → EL로 평가해서 값으로 치환  (\${...} 는 "건드리지 마"라서 통과)
  ▼
완성된 HTML/JS 텍스트  →  ② 브라우저로 전송
  ▼
  ③ 브라우저: 남아있는 ${...} 를 템플릿 리터럴로 평가 (JS 변수에서 값 찾음)
```

| 표기 | ① 서버(JSP) | 브라우저 도착 | ③ 브라우저(JS) | 결과 |
|------|-------------|----------------|----------------|------|
| `${vo.bno}` | EL로 평가 → `10` | `10` | (이미 숫자) | ✅ 서버 값 |
| `\${bno}` | "EL 아님" → `${bno}` 통과 | `` `...${bno}` `` | JS 변수 `bno` 평가 | ✅ JS 값 |
| `${bno}` (백슬래시 없이 script 안) | EL로 평가 시도 → `bno`는 서버에 없음 → 빈 값 | `` `...` `` | 평가할 게 없음 | ❌ 빈 값 |

> **`\$`의 의미** — JSP에게 "이 `${}`는 네(서버) 게 아니다, 평가하지 말고 그대로 흘려보내라". 백슬래시는 출력에서 **제거**되고 `${}`만 브라우저로 간다.

### 올바른 패턴 — 서버 값은 JS 변수로 한 번 받아두기

```jsp
<script>
    const bno = '${vo.bno}'                          // ① 서버가 값 주입 → bno = '10'
    actionForm.action = `/board/modify/${bno}`        // ③ 브라우저가 JS 변수 bno 사용
</script>
```

이렇게 하면 "서버 차례(`${vo.bno}`)"와 "브라우저 차례(`${bno}`)"가 줄마다 분리돼 헷갈리지 않는다.
**한 줄 안에서 `${서버값}`을 직접 템플릿 리터럴에 넣지 말고, 위처럼 변수로 분리하는 습관**을 들이는 게 안전하다.

### 실제 사례 — 검색 버튼: 서버값과 JS변수가 한 곳에 섞임

검색 hidden input들을 JS로 조립하는데, **서버값엔 백슬래시 없이**, **JS변수엔 백슬래시**를 붙여 둘을 구분한다.

```javascript
document.querySelector(".searchBtn").addEventListener("click", (e) => {
    e.preventDefault()
    const selectValue = document.querySelector("select[name='typeSelect']").value
    const arr = selectValue.split("")            // "TWC" → ["T","W","C"]

    let str = ''
    str += `<input type='hidden' name='pageNum' value=1 >`
    str += `<input type='hidden' name='amount' value=${cri.amount} >`   // ★ 서버값: 백슬래시 X → JSP가 10으로 치환
    for (const type of arr) {
        str += `<input type='hidden' name='types' value=\${type}>`      // ★ JS변수: 백슬래시 O → 브라우저 JS가 채움
    }
    const keywordValue = document.querySelector("input[name='keywordInput']").value
    str += `<input type='hidden' name='keyword' value='\${keywordValue}'>`  // ★ JS변수: 백슬래시 O

    actionForm.innerHTML = str
    actionForm.submit()          // ⚠️ 조립만 하고 submit()을 빠뜨리면 검색이 안 된다
})
```

| 코드 | 무엇 | 백슬래시 | 채우는 주체 |
|------|------|:-------:|------|
| `${cri.amount}` | 서버 EL 값 | **없음** | JSP(서버) |
| `\${type}` · `\${keywordValue}` | JS 변수 | **있음** | 브라우저 JS |

> 헷갈리면 위 "변수로 분리" 패턴처럼 서버값(`${cri.amount}`)도 `const amount = ${cri.amount}` 로 한 번 받아두고 `${amount}`로 쓰면 한 줄에 한 주체만 남아 더 안전하다.
> **흔한 실수**: `actionForm.innerHTML = str` 뒤 `actionForm.submit()` 누락 → 클릭해도 아무 일도 안 일어남. (8-4 폼 제출 참고)

## 3. 함수 — 선언식 · 표현식 · 화살표 함수

"함수를 만든다"는 똑같은데 **어떻게 적느냐**가 다른 것이다. 세 방식의 차이는 ① 호이스팅과 ② `this` 두 가지다.

### ① 함수 선언식 (declaration)

```javascript
function add(a, b) { return a + b }
```
- `function 이름(...)` 형태로 **이름 붙여** 선언.
- **호이스팅 통째로 됨** → 정의보다 위에서 호출해도 동작.
```javascript
add(1, 2)                               // ✅ 됨 (아래에 정의돼 있어도)
function add(a, b) { return a + b }
```

### ② 함수 표현식 (expression)

```javascript
const add = function (a, b) { return a + b }   // 함수를 '값'처럼 변수에 담음
```
- 함수를 값처럼 만들어 변수에 담는다(`function`에 이름 없음 = 익명).
- `const`에 담겼으니 **선언 전엔 못 씀**(TDZ → 10장).
```javascript
add(1, 2)                                       // ❌ ReferenceError
const add = function (a, b) { return a + b }
```

### ③ 화살표 함수 (arrow)

```javascript
const add  = (a, b) => { return a + b }
const add2 = (a, b) => a + b      // 한 줄 return이면 {} 와 return 생략 가능
const sq   = x => x * x           // 인자 1개면 () 도 생략 가능
```
- `function` 없이 `=>`로 짧게 쓰는 표현식. 호이스팅은 ②와 같다.
- **추가로 `this`가 다르게 동작**한다(아래 핵심).

| | 적는 법 | 호이스팅 | `this` |
|---|---|:---:|---|
| ① 선언식 | `function f(){}` | 통째로 O | 호출 방식따라 |
| ② 표현식 | `const f = function(){}` | X(TDZ) | 호출 방식따라 |
| ③ 화살표 | `const f = () => {}` | X(TDZ) | **바깥 것 그대로** |

> ①②는 `this`가 같고, **③ 화살표만 `this`가 다르다.** 호이스팅은 반대로 ②③가 같다.

### 핵심 차이 — `this`

`this`는 "**지금 이 함수가 누구의 것이냐**"를 가리키는데, 일반 함수와 화살표가 이걸 정하는 방식이 정반대다.

- **일반 함수** : `this`는 **"누가 이 함수를 불렀느냐(호출 방식)"**에 따라 그때그때 정해진다.
- **화살표 함수** : **자기만의 `this`가 없다.** `this`를 만나면 **자기를 감싼 바깥 코드의 `this`**를 그대로 가져다 쓴다(호출 방식 무시).

```javascript
// this 로 그 요소를 쓰고 싶으면 → 일반 함수
btn.addEventListener("click", function () {
  this;   // 👉 btn  (브라우저가 'btn을 this로' 지정해 불러줌)
})

// 화살표는 호출 방식을 무시하고 바깥 this 사용
btn.addEventListener("click", () => {
  this;   // 👉 btn 아님! 감싼 바깥의 this (보통 window)
})
```

```
일반 함수: "누가 불렀어?" → 브라우저가 btn으로 불러줌 → this = btn  ✅
화살표   : "누가 부르든 상관없어, 내 바깥 거 쓸래"   → this = window ❌(btn 아님)
```

| | 일반 함수 `function` | 화살표 함수 `=>` |
|---|---|---|
| `this` | **호출될 때** 결정 (이벤트면 그 요소) | **자신을 감싼 바깥**의 `this` 그대로 |
| `arguments` 객체 | 있음 | 없음 |

### 실무에서 어떻게 고르나

```javascript
// ✅ this 로 그 요소를 쓴다 → 일반 함수
btn.addEventListener("click", function () {
  this.classList.add("active")            // this = btn
})

// ✅ this 안 쓰고 e.currentTarget 으로 대신 → 화살표가 깔끔
btn.addEventListener("click", (e) => {
  e.currentTarget.classList.add("active") // currentTarget = 리스너 붙은 요소 (= 일반함수의 this와 같음)
})
```

> **정리 — `this`를 쓸 거면 일반 함수, 안 쓰고 `e.currentTarget`으로 대신할 거면 화살표.**
> 헷갈리면 그냥 **`e.target`/`e.currentTarget`을 쓰고 화살표로 통일**하는 게 실수가 적다(`this` 함정 회피).
> 핸들러에서 `this`를 안 쓰고 `e`(이벤트 객체)만 쓴다면 일반/화살표 어느 쪽이든 동작은 같다.

## 4. 비교 `==` vs `===` 와 truthy / falsy

```javascript
1 == '1'    // true  ← 타입을 강제로 맞춰서 비교 (느슨한 비교)
1 === '1'   // false ← 타입까지 같아야 true (엄격한 비교) ✅ 기본으로 이걸 써라
```

**falsy(거짓 취급) 값 6가지** — 이것 말고는 전부 truthy:
```
false · 0 · '' (빈 문자열) · null · undefined · NaN
```

```javascript
const result = '${result}'   // 서버가 값 없으면 '' (빈 문자열 → falsy)
if (result) {                // 값이 있을 때만 (빈 문자열이면 통과 안 함)
  myModal.show()
}
```

> **주의 — `'0'`, `'false'`, `[]`, `{}` 는 전부 truthy.** 문자열 `'0'`은 빈 문자열이 아니라서 참이다.

## 5. 구조 분해 · 스프레드 · 나머지(rest)

```javascript
// 구조 분해 (destructuring) — 객체/배열에서 값만 빼오기
const { pageNum, amount } = cri          // cri.pageNum, cri.amount 를 한 번에
const [first, second] = [10, 20]

// 스프레드 (...) — 펼치기
const arr = [...nodeList]                // 유사배열 → 진짜 배열 (forEach/map 쓰려고)
parent.prepend(...items)                 // 여러 요소를 한 번에

// 나머지 매개변수 (...) — 모으기
function sum(...nums) { return nums.reduce((a, b) => a + b, 0) }
```

> **활용 포인트** — `getElementsByClassName`/`querySelectorAll` 결과는 배열이 아니다(HTMLCollection/NodeList).
> `[...el]`로 펼쳐야 `map`/`filter` 같은 배열 메서드를 쓸 수 있다.

## 6. 옵셔널 체이닝 `?.` 와 널 병합 `??`

```javascript
// ?. — 중간이 null/undefined면 에러 대신 undefined 반환
const bno = target?.dataset?.bno         // target이 없어도 터지지 않음

// ?? — 왼쪽이 null/undefined일 때만 오른쪽 사용 (0, '' 는 그대로 둠)
const page = pageNum ?? 1                // pageNum이 0이면 0 유지 (|| 와 다름!)
```

> **`??` vs `||` 주의** — `||`는 falsy(`0`, `''`)면 전부 오른쪽으로 넘어간다.
> 페이지 번호 `0`을 유효한 값으로 둬야 한다면 `||` 말고 `??`를 써야 한다.

## 7. 형변환 — 문자열 ↔ 숫자 (DOM에서 자주 실수)

**DOM에서 읽은 값은 거의 다 문자열이다.**

```javascript
const targetPage = target.getAttribute("href")   // "2" (문자열!)
input.value                                        // 항상 문자열

Number("2")        // 2
parseInt("2px")    // 2   (앞쪽 숫자만)
+"2"               // 2   (단항 + 로 빠르게 변환)
String(2)          // "2"
```

> **주의 — `+`는 문자열이 하나라도 있으면 "이어붙이기"가 된다.**
> ```javascript
> "2" + 1   // "21"  (문자열 연결!)
>  2  + 1   // 3
> ```
> 서버로 보낼 땐 문자열이어도 Spring이 알아서 숫자로 바꿔주지만(9장), JS 안에서 계산할 땐 먼저 `Number()`로 변환할 것.

## 8. 폼(Form) 동작 — 제출은 어떻게 일어나는가

폼 데이터를 서버로 보내는 방법은 두 가지다.

```
① 기본 제출   : 사용자가 submit 버튼 클릭 / 입력 후 Enter  → 브라우저가 자동 제출
② JS 제출     : 코드에서 form.submit() 호출                → 우리가 직접 제출
```

### 8-1. `<button type="...">` 3종 — 기본값이 함정

| type | 동작 |
|------|------|
| `submit` | **자기가 속한 form을 제출** (type 생략 시 기본값!) |
| `reset` | 폼 입력값 초기화 |
| `button` | 아무 기본 동작 없음 (JS 전용 버튼) |

> ⚠️ **`<button>`의 type을 생략하면 자동으로 `submit`이다.** 그래서 JS로만 처리할 버튼인데 type을 안 적으면, 클릭할 때마다 폼이 멋대로 제출돼서 페이지가 새로고침된다.

### 8-2. 버튼은 "자기가 속한 form"을 제출한다 — 폼 2개 사례

한 화면에 폼이 **2개** 있다고 하자.

```jsp
<form id="actionForm" action="/board/modify" method="post">
    ...
    <button type="submit" class="btnList">LIST</button>     ← actionForm 안에 있음
</form>

<form id="listForm" action="/board/list" method="get"> ... </form>   ← LIST가 가고 싶은 곳
```

LIST 버튼은 `#actionForm` 안에 있어서, **그냥 두면 `actionForm`(=POST /board/modify)을 제출**한다.
하지만 우리가 원하는 건 `#listForm`(=GET /board/list) 제출이다.

### 8-3. `preventDefault()` 가 폼에서 필수인 이유 (동작과정)

LIST 클릭 시 일어나는 두 동작이 **충돌**한다:

| | 누가 | 어디로 |
|---|---|---|
| ① 기본 동작 | 버튼이 속한 `actionForm` 제출 | POST `/board/modify` ❌ |
| ② 내 핸들러 | `listForm.submit()` | GET `/board/list` ✅ |

```javascript
document.querySelector(".btnList").addEventListener("click", function (e) {
    e.preventDefault()      // ① 기본 제출(actionForm) 취소 → 이게 없으면 엉뚱한 곳으로 감
    e.stopPropagation()     // 이벤트가 부모로 버블링되는 것 차단 (여기선 안전장치)
    listForm.submit()       // ② 내가 원하는 폼만 제출
})
```

> **동작 순서**: 클릭 → 핸들러에서 `preventDefault()`로 브라우저 기본 제출을 꺼버림 → `listForm.submit()`만 살아남아 `/board/list`로 이동.
> `preventDefault()`가 없으면 ①번 기본 제출이 먼저 실행돼 페이지가 `/board/modify`로 떠나버린다 → "리스트로 가는 게 안 된다"의 정체.

`<a href>` 링크도 같은 원리다. 예를 들어 페이지네이션이 `<a href="2">`라면 기본 동작이 "그 href로 이동"인데, 폼 제출을 하고 싶다면 `preventDefault()`로 링크 이동을 막는다.

> 📌 `preventDefault()`(기본 동작 차단)와 `stopPropagation()`(버블링 차단)은 **다른 것**이다. 헷갈림 주의.

### 8-4. `form.submit()` 으로 action/method 동적 변경

```javascript
// 같은 폼을 상황에 따라 다른 곳으로 보냄
actionForm.action = `/board/modify/${bno}`   // 보낼 주소 바꾸고
actionForm.method = 'post'                    // 메서드 바꾸고
actionForm.submit()                           // 제출
```

> ⚠️ **함정 — input의 name이 `submit`이면 `form.submit()`이 깨진다.**
> `<input name="submit">`이 있으면 `form.submit`이 그 input을 가리켜서, 함수 호출 시 `form.submit is not a function` 에러가 난다. 버튼/인풋 name에 `submit`, `action` 같은 폼 속성명을 쓰지 말 것.

### 8-5. GET 폼 vs POST 폼

| | GET | POST |
|---|---|---|
| 데이터 위치 | URL 쿼리스트링 `?a=1&b=2` | HTTP body (URL에 안 보임) |
| 용도 | 조회·검색(목록, 페이징) | 변경(등록·수정·삭제) |
| 예 | `listForm` (목록 조회) | `actionForm` (수정/삭제) |

### 8-6. hidden input — 화면에 안 보이게 상태 넘기기

```jsp
<form id="listForm" action="/board/list" method="get">
    <input type="hidden" name="pageNum" value="${cri.pageNum}">   <!-- 서버가 값 주입 -->
    <input type="hidden" name="amount" value="${cri.amount}">
</form>
```

> 페이지를 오가도 "몇 페이지였는지"를 유지하려고 hidden으로 들고 다닌다. `value="${cri.pageNum}"`는 **서버 EL**이 채워준다(2장 ① 단계).

## 9. 폼 `name` → 서버 객체로 자동 바인딩 (Spring 연결점)

폼을 제출하면 각 입력의 `name`이 요청 파라미터가 되고, **Spring이 같은 이름의 setter를 찾아 객체에 담아준다.**

```
<input name="pageNum" value="2">          (브라우저)
        │ 제출 → /board/list?pageNum=2&amount=10
        ▼
@GetMapping("/list")
public void list(@ModelAttribute("cri") Criteria cri) { ... }   (서버)
        │
        ▼  Spring: new Criteria() → setPageNum(2) → setAmount(10)
```

| form `name` | Criteria setter | 결과 |
|-------------|-----------------|------|
| `pageNum` | `setPageNum(int)` | ✅ |
| `amount` | `setAmount(int)` | ✅ |

> **연결 고리는 "이름이 같다"는 것뿐.** `name`과 setter 이름만 맞으면 `@ModelAttribute`가 객체를 만들어 채운다.
> 값은 문자열("2")로 와도 Spring이 `int`로 변환해준다.

## 10. 호이스팅(Hoisting)

**선언이 자기 스코프의 맨 위로 끌어올려진 것처럼 동작하는 것.**
JS 엔진은 코드를 실행하기 **전에** 먼저 훑으면서 "어떤 변수·함수가 선언돼 있나"를 미리 등록한다. 그래서 선언보다 위에서 써도 동작하는(또는 에러 나는) 일이 생긴다.

```javascript
// ── var: 선언은 끌어올려지고 undefined로 초기화 ──
console.log(a)   // undefined  (에러 아님 — 선언은 이미 위에 등록됨, 값만 아직 없음)
var a = 10
console.log(a)   // 10

// ── 함수 선언식: 통째로 끌어올려짐 → 정의보다 위에서 호출 가능 ──
f()              // ✅ "hi"
function f() { console.log("hi") }

// ── let/const: 끌어올려지긴 하나 TDZ(초기화 전 구역)에 갇힘 ──
console.log(b)   // ❌ ReferenceError (var처럼 undefined로 안 봐줌)
let b = 20
```

| 대상 | 호이스팅 | 선언 전 접근 시 |
|------|:--------:|------------------|
| `var` | O (undefined로 초기화) | `undefined` — 조용히 넘어가 **버그 유발** |
| 함수 선언식 `function f(){}` | O (통째로) | 정상 호출됨 |
| `let` / `const` | O (단, TDZ) | **ReferenceError** ✅ 실수를 잡아줌 |
| 함수 표현식 `const f = () => {}` | const 규칙 따름 | ReferenceError |

> **핵심** — `let`/`const`도 호이스팅은 되지만 `var`처럼 `undefined`로 슬쩍 봐주지 않고 **에러를 낸다.** "선언 전에 쓰는 실수"를 잡아주므로 더 안전하다. (1장에서 말한 TDZ가 이것)

## 11. 스크립트 로딩 — `</body>` 직전 vs `defer` / `async`

JS는 위에서 아래로 읽고, `<script>`를 만나면 **HTML 파싱을 멈추고** 스크립트를 받아 실행한다(블로킹). 그래서 위치가 중요하다.

- `<head>`에 그냥 두면 → DOM이 아직 없어서 `querySelector`가 `null`
- 다운로드 동안 화면 그리기가 멈춤

```
일반 <script>  : [HTML파싱 ▮멈춤▮ 다운로드+실행 ▮재개▮ ...]     ← 파싱을 막음
async          : [HTML파싱 ━━━━━] 받는 즉시 ▮멈추고 실행▮         ← 순서 보장 X
defer          : [HTML파싱 ━━━━━━━━━] 파싱 끝난 뒤 순서대로 실행   ← 순서 보장 O
                  (async·defer 모두 다운로드는 파싱과 병렬)
```

| | 다운로드 | 실행 시점 | 순서 보장 | 용도 |
|---|---|---|:---:|---|
| 일반 | 파싱 멈추고 | 즉시 | O | (옛날 방식) |
| `async` | 병렬 | 받는 즉시(파싱 중단) | ✕ | 독립 스크립트(분석·광고) |
| `defer` | 병렬 | **HTML 파싱 끝난 뒤**, 순서대로 | O | **대부분의 앱 스크립트** |

**예전 방식** — 스크립트를 맨 아래로:
```html
<body>
  ... 내용 ...
  <script src="app.js"></script>   <!-- DOM 다 만들어진 뒤 실행 -->
</body>
```

**요즘 권장** — `<head>`에 두되 `defer`:
```html
<head>
  <script src="app.js" defer></script>   <!-- 미리 받아두고 실행은 파싱 후 -->
</head>
```
- DOM 완성 후 실행 → `querySelector` 안전 (맨 아래 두는 것과 같은 효과)
- 다운로드는 파싱과 병렬 → 더 빠름
- 여러 스크립트 **실행 순서 보장** (`async`는 안 됨)
- `<script type="module">`은 **자동으로 defer처럼** 동작

> **인라인 스크립트라면?** `<script>`를 `</body>` 직전(footer 다음)에 **인라인**으로 두면 옛날 방식으로 안전하게 동작한다.
> ⚠️ `defer`/`async`는 **외부 파일(`<script src>`)에만** 적용된다. **인라인 스크립트엔 무시**되므로, 인라인이면 위치(맨 아래)로 해결하는 게 맞다.

## 주의점 모음 (자주 실수하는 것)

1. **`const` ≠ 불변** — 객체 내부는 바뀐다. 재할당만 막힐 뿐.
2. **`${}` 두 개가 똑같이 생겼다** — script 안에서 서버값은 `${vo.bno}`, JS값은 `\${jsVar}`. 변수로 분리해라.
3. **`<button>` type 생략 = submit** — JS 전용 버튼은 `type="button"` 명시.
4. **버튼은 자기 부모 form을 제출** — 다른 폼 보낼 거면 `preventDefault()` 먼저.
5. **`preventDefault` ≠ `stopPropagation`** — 기본 동작 차단 vs 버블링 차단.
6. **DOM에서 읽은 값은 문자열** — 계산 전 `Number()` 변환. `+`는 문자열이면 연결.
7. **`==` 대신 `===`** — 타입 강제변환 함정 피하기.
8. **`||` vs `??`** — `0`/`''`을 유효값으로 둘 거면 `??`.
9. **input name에 `submit` 금지** — `form.submit()`이 깨진다.
10. **HTMLCollection/NodeList는 배열 아님** — `[...el]`로 변환 후 `map`/`filter`.
11. **`var`는 선언 전 접근해도 `undefined`** — 버그 유발. `let`/`const`(TDZ)가 에러로 잡아준다.
12. **`defer`/`async`는 인라인 `<script>`엔 무시** — 외부 파일에만 적용. 인라인이면 위치로 해결.
