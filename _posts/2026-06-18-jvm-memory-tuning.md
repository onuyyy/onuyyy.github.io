## 0. 출발점 — 왜 이 얘기가 나왔나

`docker stats`에서:
```
NAME            MEM USAGE / LIMIT
my-app  376.8MiB / 512MiB   (73.6%)
```
- app 컨테이너가 한도 512MB의 73.6%를 쓰고 있어서 "JVM이 메모리 누수/폭주하는 것 아닌가?" 의심.
- 결론: **아니었음.** JVM은 정상이고, 376MB는 정상 footprint였다. (아래 4번)

## 1. JVM이란 — Java는 가상머신 위에서 돈다

Java 프로그램은 OS 위에서 바로 도는 게 아니라 **JVM(Java Virtual Machine)이라는 가상머신** 위에서 실행된다.
→ 그래서 JVM이 메모리를 "어떻게 잡고 쓰느냐"가 곧 컨테이너 메모리 사용량을 결정한다.

JVM 메모리 사용의 3가지 특징:
1. **힙(Heap)을 미리 확보** — 객체 담을 공간을 시작 시 큼직하게 잡아둠
2. **GC(Garbage Collector)가 자동 청소** — 안 쓰는 객체를 알아서 치움
3. **청소해도 OS에 잘 안 돌려줌** — "또 쓸지 모르니 들고 있자" → 내버려 두면 메모리가 부푼다

## 1.5 JVM 아키텍처 & 동작 방식 — "코드가 실행되기까지"

### 전체 구조 (3대 구성요소)
JVM은 크게 **클래스 로더 → 런타임 데이터 영역 → 실행 엔진** 세 덩어리로 돈다.

```
   소스             컴파일(javac)        실행(JVM)
 Hello.java  ───────────────────────▶  Hello.class (바이트코드)
                                              │ 로딩
        ┌─────────────────────────────────────▼──────────────────────────────┐
        │                          J V M                                      │
        │                                                                     │
        │  ① 클래스 로더(Class Loader)                                         │
        │     Loading → Linking(검증·준비·해석) → Initialization               │
        │     .class 를 읽어 메서드 영역에 클래스 정보 적재                       │
        │                              │                                       │
        │                              ▼                                       │
        │  ② 런타임 데이터 영역(Runtime Data Areas)  ← 메모리(2번 섹션 상세)     │
        │     [공유] 힙 · 메서드영역(메타스페이스)                               │
        │     [스레드별] JVM 스택 · PC 레지스터 · 네이티브 스택                  │
        │                              │                                       │
        │                              ▼                                       │
        │  ③ 실행 엔진(Execution Engine)                                       │
        │     ├─ 인터프리터  : 바이트코드를 한 줄씩 해석 실행 (초기엔 빠른 시작)  │
        │     ├─ JIT 컴파일러 : 자주 도는 코드(핫스팟)를 기계어로 컴파일 → Code   │
        │     │                Cache 에 저장 → 다음부턴 네이티브 속도            │
        │     └─ GC          : 힙에서 안 쓰는 객체 회수                         │
        │                              │                                       │
        │  ④ JNI ──▶ 네이티브 메서드 라이브러리(OS, C/C++ .so/.dll)            │
        └─────────────────────────────────────────────────────────────────────┘
```

### 동작 흐름 한눈에
```
.java ─(javac)→ .class(바이트코드) ─(클래스로더)→ 메서드영역 적재
   → 인터프리터가 실행 시작 → 자주 도는 부분은 JIT가 기계어로 컴파일(Code Cache)
   → 실행 중 객체는 힙에, 호출 프레임은 스택에 → 안 쓰는 객체는 GC가 회수
```

| 구성요소 | 역할 | 메모리와의 연결 |
|---|---|---|
| **클래스 로더** | `.class` 읽어 적재·검증·초기화 | 적재 결과 → **메서드 영역(메타스페이스)** |
| **인터프리터** | 바이트코드 한 줄씩 실행 | 초기 구동·드물게 도는 코드 |
| **JIT 컴파일러** | 핫스팟을 기계어로 | 결과를 **Code Cache**(non-heap)에 |
| **GC** | 힙의 쓰레기 객체 회수 | **힙**을 대상으로 동작 |
| **JNI / 네이티브** | OS·C 라이브러리 호출 | **네이티브 메서드 스택** |

> 왜 "한 번 작성하면 어디서나 실행(WORA)"? → 소스를 **OS별 기계어가 아니라 바이트코드(.class)**로 컴파일하고,
> 그 바이트코드를 **각 OS에 맞는 JVM**이 받아 실행하기 때문. 플랫폼 차이는 JVM이 흡수한다.
>
> 왜 "처음엔 느리다가 점점 빨라지나"? → 시작은 **인터프리터**(컴파일 대기 없이 바로 실행),
> 자주 도는 코드는 **JIT**가 기계어로 바꿔 캐싱하므로 워밍업 후 속도가 붙는다.

## 2. JVM 메모리 구조 — "힙이 전부가 아니다" (핵심)

JVM이 쓰는 메모리 = **힙(Heap) + 힙이 아닌 것(Non-heap)**

```
컨테이너가 쓰는 전체 메모리
├─ 힙(Heap)          ← 우리가 만든 객체들. -Xmx / MaxRAMPercentage 로 조절
└─ 힙 외(Non-heap)   ← 아래 합. JVM이 살아 돌아가는 데 필요한 부대 비용
   ├─ 메타스페이스(Metaspace)  클래스 정보. Spring은 클래스가 많아 큼
   ├─ 스레드 스택(Thread Stack) 스레드 수 × 스택 크기
   ├─ JIT 컴파일된 코드(Code Cache)
   └─ 네이티브 메모리 / JVM 자체
```

> **가장 흔한 오해:** "힙 = JVM 메모리 전부"가 아니다.
> 그래서 힙을 컨테이너 한도의 100%로 잡으면 → non-heap 들어갈 자리가 없어 한도 초과 → **OOMKilled**.
> 그래서 힙은 보통 한도의 **70~75%**까지만 잡는다(나머지는 non-heap 몫).

### 실측 예
```
컨테이너 사용 376MB = 힙(최대 128MB) + 힙외(~248MB)
```
- 힙은 128MB뿐인데 전체가 376MB인 이유 = **non-heap ~248MB**(Spring Boot의 클래스·스레드·네이티브).
- Spring Boot 앱은 원래 이 정도 non-heap을 쓴다 → **비정상 아님.**

### 런타임 데이터 영역(Runtime Data Areas) — 구조 상세
JVM이 잡는 메모리를 정식 명칭으로 펼치면:

```
JVM 런타임 데이터 영역
├─ [공유] 힙(Heap)              객체·배열. 모든 스레드가 같이 씀     ┐ 힙
│                               GC 대상. -Xmx / MaxRAMPercentage   ┘ (제어 가능)
│
├─ [공유] 메서드 영역(Method Area)  클래스 메타·상수풀·static       ┐
│         = Java 8+에선 메타스페이스(Metaspace), 네이티브 메모리에 위치 │
├─ [스레드별] JVM 스택(Stack)    메서드 호출 프레임·지역변수          │ non-heap
├─ [스레드별] PC 레지스터         그 스레드가 지금 실행 중인 위치       │ (대부분
└─ [스레드별] 네이티브 메서드 스택  JNI 등 네이티브 호출용             ┘  자동)
   + (데이터 영역 밖) Code Cache(JIT 컴파일 결과), JVM 엔진 자체 네이티브
```

| 영역 | 공유 vs 스레드별 | 무엇을 담나 | 크기를 키우는 요인 |
|---|---|---|---|
| 힙 | 전 스레드 공유 | new 한 객체·배열 | 요청량·캐시·세션 |
| 메서드 영역(메타스페이스) | 공유 | 로딩된 클래스 정보 | **클래스 수**(Spring은 많음) |
| JVM 스택 | 스레드마다 1개 | 호출 프레임·지역변수 | **스레드 수** × 스택 깊이 |
| PC 레지스터 | 스레드마다 1개 | 실행 위치 포인터 | (매우 작음) |
| 네이티브 스택 | 스레드마다 1개 | 네이티브 호출 | 네이티브 사용량 |
| Code Cache | 공유 | JIT 기계어 | 실행 핫스팟 양 |

→ 그래서 **스레드를 많이 띄우는 웹앱(톰캣 스레드풀)**은 스택·메타스페이스가 함께 커져 non-heap이 커진다.

### ⭐ 핵심: 힙은 "제어", non-heap은 "비워두기" (헷갈림 1순위)
```
mem_limit: 512m  (컨테이너 벽)
┌──────────────────────────────────────────────┐
│ 힙 (Heap)                  non-heap            │
│ ████████████████░░░░░░░░░│▒▒▒▒▒▒▒▒             │
│ MaxRAMPercentage=70       "남겨둔 30% 자리"      │
│ → 358MB로 내가 *막음*      → 메타스페이스·스택이    │
│   (진짜 손잡이 있음)         앱 상황 따라 *알아서* 채움 │
└──────────────────────────────────────────────┘
```
- **힙** = `-Xmx` / `MaxRAMPercentage`로 **상한을 직접 건다.** 손잡이가 있음.
- **non-heap** = "30%로 설정"하는 게 **아님.** 클래스 수·스레드 수에 따라 **스스로 늘어남.**
  내가 하는 건 "non-heap이 늘어날 자리를 침범당하지 않게 **힙을 70%로 막아 30%를 비워두는 것.**"
- 그래서 힙을 100%로 잡으면 → non-heap이 늘 자리가 없어 한도 초과 → **OOMKilled.**

> 한 줄: **"JVM 메모리 = 힙"이 아니라, 힙은 데이터 영역의 일부.**
> 튜닝에서 내가 정하는 건 힙 상한뿐이고, non-heap은 "남겨두는 여유"로 관리한다.

## 3. 컨테이너에서의 핵심 문제 — "JVM이 자기 방 크기를 착각한다"

### 옛날 JVM(Java 8 초기)의 문제
JVM이 힙 크기를 정할 때 기준으로 **호스트(서버) 전체 RAM**을 봤다.

```
실제:                          JVM의 착각(구버전):
┌──────────────────┐          "서버 RAM이 1.9GB네?
│ EC2 전체 RAM 1.9GB │           그럼 그 25%인 ~475MB까지
│  ┌────────────┐  │   →        힙으로 써도 되겠다"
│  │컨테이너 512MB │  │
│  │  JVM        │  │          ← 근데 컨테이너 벽은 512MB뿐!
│  └────────────┘  │             힙 475 + non-heap → 512 초과 → OOMKilled
└──────────────────┘
```

→ `mem_limit 512m`(컨테이너 벽)은 있는데, **JVM이 그 벽을 못 보고** 호스트 RAM 기준으로 힙을 키우다 죽는 것.

### 해결: Java 10+ 의 컨테이너 자동 인식
- **Java 10부터 `UseContainerSupport`가 기본 ON** → JVM이 컨테이너 한도(cgroup)를 직접 읽어 그 기준으로 힙을 계산한다.
- **Java 21**(예: Temurin 21)에선 이 착각 문제가 **애초에 발생하지 않는다.**

## 4. 진단 방법 — "JVM이 컨테이너를 제대로 인식하나?"

### 명령
```bash
# 1) Java 맞는지 + 버전 (10+ 이어야 자동 인식)
docker exec <컨테이너> java -version

# 2) JVM이 잡은 힙 최대치 확인 (바이트 단위)
docker exec <컨테이너> java -XX:+PrintFlagsFinal -version 2>/dev/null | grep -i maxheapsize
```

### 실측 결과 해석
```
openjdk version "21.0.11" ... Temurin-21.0.11   ← Java 21, 자동 인식 OK
MaxHeapSize = 134217728                          ← 128MiB
```

**판독:**
```
134217728 byte ÷ 1024 ÷ 1024 = 128 MiB
512MB(컨테이너 한도) × 25%(Java 기본 힙 비율) = 128MB   ← 정확히 일치!
```
- 힙이 **128MB(=512의 25%)** → JVM이 "내 방은 512MB"라고 **정확히 인식 중**. ✅
- 만약 호스트(1.9GB)를 봤다면 `1.9GB × 25% ≈ 475MB`가 나왔을 것. → 그게 아니므로 **착각 없음.**

### 판독 기준표
| MaxHeapSize | 의미 | 조치 |
|---|---|---|
| 한도의 ~25% (기본값) | 컨테이너 정상 인식 | 보통 그대로 OK |
| 한도의 70%~꽉 참 | 힙이 커서 non-heap 터질 위험 | 힙 낮추기 검토 |
| 호스트 RAM 기준 큰 값 | **착각 중(구버전)** | 튜닝 필수 |

## 5. 튜닝이 정말 필요할 때 — 하는 법

> 이 사례는 튜닝 불필요였지만, 필요한 상황을 위해 정리.
> 튜닝 = "한도를 새로 만드는 것"이 아니라 "JVM이 한도를 알아듣게 / 힙 크기를 조정하는 것".

### 방법 1: MaxRAMPercentage (요즘 권장 — 비율)
```yaml
# docker-compose.yml
services:
  app:
    mem_limit: 512m
    environment:
      - JAVA_TOOL_OPTIONS=-XX:MaxRAMPercentage=70.0   # 512의 70% ≈ 358MB까지 힙
```
- 컨테이너 크기를 바꿔도 비율로 자동 추종 → 컨테이너 친화적.
- 기본값(25%)이 너무 보수적이라 힙을 더 주고 싶을 때 올린다. **단 non-heap 몫(25~30%)은 꼭 남길 것.**

### 방법 2: -Xmx / -Xms (고전 — 절대값)
```yaml
    environment:
      - JAVA_TOOL_OPTIONS=-Xms256m -Xmx384m
```
- `-Xmx384m` = 힙 최대 384MB / `-Xms256m` = 힙 시작 256MB
- `-Xms`와 `-Xmx`를 **같은 값**으로 두면 실행 중 힙을 늘렸다 줄였다 하는 비용이 없어 성능 안정.

### ⚠️ 절대 하면 안 되는 것
- 힙을 컨테이너 한도의 100%로 잡기 → non-heap 자리 없음 → OOMKilled.
- "여유 있어 보이니 힙 왕창 늘리기" → 예: 힙 358 + non-heap 248 = 606 > 512 → 죽음.

## 6. "튜닝이 필요한지" 판단 신호

힙을 **늘려야** 하는 신호 (힙이 너무 작아 GC 폭주):
- 응답이 주기적으로 느려짐(Full GC 멈춤)
- GC 로그에 Full GC가 자주 / 오래
```bash
# GC 동작 관찰
docker exec <컨테이너> jstat -gcutil 1 2000 10   # 2초 간격 10번, O(Old) 사용률·FGC 횟수 관찰
```

힙을 **줄여야** 하는 신호 (non-heap 터짐):
- 힙은 여유인데 컨테이너가 OOMKilled → non-heap이 한도 초과 → 힙 비율 낮추거나 한도 늘리기

대부분의 경우: **Java 10+ 자동 인식이면 손대지 말 것.** 멀쩡한 걸 건드리면 오히려 망가진다.

## 7. 요약 (한 장)

| 질문 | 답 |
|---|---|
| JVM 메모리 = 힙? | ❌ 힙 + non-heap(메타스페이스·스택·네이티브). 힙은 일부일 뿐 |
| 컨테이너에서 왜 죽나 | 구버전 JVM이 호스트 RAM 기준 힙을 잡아 한도 초과 → OOMKilled |
| Java 21은? | 컨테이너 자동 인식(UseContainerSupport 기본 ON) → 착각 없음 |
| 힙 최대치 확인 | `java -XX:+PrintFlagsFinal -version \| grep MaxHeapSize` |
| 기본 힙 비율 | 컨테이너 한도의 **25%** |
| 튜닝 방법 | `MaxRAMPercentage`(비율, 권장) 또는 `-Xmx/-Xms`(절대값) |
| 힙 100% 잡으면? | non-heap 자리 없어 OOMKilled. **70~75%까지만** |
| 결론 | MaxHeapSize=128MB=512의 25% → 정상 인식. **튜닝 불필요** |

> 핵심 교훈: **"메모리 많이 쓴다 ≠ JVM 문제."** 진단(MaxHeapSize 확인) 먼저, 튜닝은 그 다음.
> 그리고 진짜 원인은 JVM이 아니라 **서버가 작고 한도 합이 오버커밋**이었다 → 서버 운영 노트 참고.
